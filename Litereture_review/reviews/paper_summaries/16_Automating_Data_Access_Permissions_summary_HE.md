## 1. הבעיה המרכזית — למה הרשאות קלאסיות לא מתאימות לסוכני AI?

### תובנת המפתח

סוכני AI (Agentic Systems) פועלים באופן אוטונומי בשם המשתמש — מזמינים טיסות, ניגשים לאנשי קשר, קוראים מסמכים. הבעיה: **מודלים קלאסיים של הרשאות (permissions) — כמו אלה שבטלפון או באפליקציות — לא מתאימים לפרדיגמת הביצוע האוטומטי של סוכנים**. המאמר מציע **ניהול הרשאות אוטומטי** (Automated Permission Management) — מערכת שלומדת את העדפות המשתמש ומחליטה בשמו אילו נתונים לשתף עם הסוכן.

### מטאפורה: "העוזר האישי החדש במשרד"

דמיינו שהעסקתם עוזר אישי חדש (AI Agent) במשרד:

```
בעולם הישן (אפליקציות מובייל):
┌───────────────────────────────────────────────┐
│  עוזר מבקש מפתח לחדר הצילום (install-time)   │
│  → אתם נותנים / לא נותנים                    │
│  → העוזר תמיד עושה את אותו הדבר              │
│  → הבקשות צפויות מראש                        │
└───────────────────────────────────────────────┘

בעולם החדש (סוכני AI):
┌───────────────────────────────────────────────┐
│  עוזר אומר: "אני צריך גישה לכרטיס האשראי    │
│  שלך, לאנשי הקשר, לתיק הרפואי, ולמסמכי       │
│  הנסיעות — כדי להזמין לך טיסה"               │
│                                               │
│  → כל משימה דורשת נתונים שונים                │
│  → אי אפשר לדעת מראש מה יצטרך               │
│  → לשאול בכל פעם = "permission fatigue"       │
│  → לתת הכל = סיכון אבטחה וחשיפת פרטיות      │
└───────────────────────────────────────────────┘
```

**הפתרון שהמאמר מציע:** לבנות "עוזר להרשאות" (Permission Assistant) שלומד את ההעדפות שלכם ומחליט אוטומטית — כמו שמזכירה ותיקה יודעת אילו מסמכים מותר לשתף ואילו לא.

## 2. רקע — מדוע הרשאות קיימות נכשלות?

### 2.1 שני סוגי הרשאות קלאסיים

| סוג | מתי | דוגמה | מגבלה לסוכני AI |
|-----|------|-------|----------------|
| **Install-time** | בעת ההתקנה | "האפליקציה צריכה גישה למצלמה" | הסוכן לא יודע מראש אילו נתונים יצטרך — ההתנהגות נקבעת ב-runtime |
| **Runtime** | בעת שימוש | "אשר גישה למיקום עכשיו" | הסוכן צריך גישה לנתונים **רבים** בכל שאילתה — שאלות חוזרות = permission fatigue |

### 2.2 מה ייחודי בסוכני AI?

```
אפליקציה רגילה:                    סוכן AI:
┌──────────────┐                   ┌──────────────────────────┐
│ מבקשת: GPS   │                   │ מבקש: שם, תאריך לידה,   │
│ פעם אחת      │                   │ כרטיס אשראי, דרכון,      │
│ תמיד אותו   │                   │ כתובת, לוח שנה, אנשי    │
│ דבר          │                   │ קשר — וזה רק לשאילתה    │
└──────────────┘                   │ אחת!                     │
                                   │ השאילתה הבאה = סט אחר!  │
                                   └──────────────────────────┘
```

**שלוש בעיות קריטיות:**

1. **לא ניתן לצפות מראש** — משאבי הנתונים נקבעים רק ב-runtime, בהתאם לשאילתת המשתמש
2. **כמות הבקשות** — כל שאילתה עשויה לדרוש מספר סוגי נתונים, מה שיוצר "עייפות הרשאות"
3. **כלים לא מהימנים** — כלי צד שלישי (third-party tools) עלולים לאסוף יותר נתונים ממה שנדרש, או להיות זדוניים

### 2.3 מודל האיומים (Threat Model)

המאמר מניח מודל איומים מעשי:

```
┌─────────────────────────────────────────────────────────────┐
│                      מודל האיומים                           │
│                                                             │
│  🤖 סוכן AI — לא זדוני, אבל עלול לטעות                    │
│     • עשוי לאסוף נתונים מיותרים בטעות                      │
│     • אמביגואיות בשפה טבעית → פרשנות שגויה                │
│     • hallucination → גישה לנתונים שלא צריך                │
│                                                             │
│  🔧 כלי צד שלישי — עלולים להיות:                           │
│     • לא מהימנים (untrustworthy)                           │
│     • זדוניים (malicious)                                  │
│     • פגועים (compromised via prompt injection)             │
│     • אוספי נתונים מופרזים                                 │
│                                                             │
│  👤 משתמש — רוצה שליטה אבל לא רוצה שיפריעו לו            │
│     • over-permissioning > under-permissioning              │
│     • מתקשה להעריך אילו נתונים באמת נדרשים                 │
│     • מושפע מהקשר, רגישות הנתון, וביטחון ב-AI              │
│                                                             │
│  🛡️ עוזר ההרשאות (Permission Assistant) — מהימן           │
│     • פרוס בשכבת ה-scaffolding של הסוכן                    │
│     • ל-LLM אין גישה ישירה לעוזר                          │
│     • מקבל החלטות הסתברותיות — עלול לטעות                  │
│     • המשתמש שומר על שליטה סופית                           │
└─────────────────────────────────────────────────────────────┘
```

## 3. מתודולוגיה — מחקר משתמשים מבוסס תרחישים (Vignette-Based User Study)

### 3.1 מטרות המחקר

שלוש מטרות מרכזיות:

1. **הבנת העדפות** — אילו גורמים משפיעים על החלטות הרשאה של משתמשים?
2. **למידה וחיזוי** — האם אפשר ללמוד את ההעדפות ולחזות אותן אוטומטית?
3. **אכיפה** — (מחוץ לתחום המאמר הזה) איך לאכוף את ההחלטות בפועל

### 3.2 עיצוב המחקר

#### מסגרת יצירת שאלות (Question Curation)

המחקרים בנו framework מבוסס LLM ליצירת תרחישים:

```
קלט:
  8 תחומים (domains) × 21 כלים (tools) × סוגי נתונים

תהליך:
  1. בוחרים תחום (למשל: Travel)
  2. מגדירים כלים רלוונטיים (Weather, Travel Booking, Calendar...)
  3. LLM מייצר שאילתות טבעיות שדורשות את הכלים
  4. 3 חוקרים מסננים ומאחדים שאילתות דומות

פלט:
  65 שאלות × 8 תחומים × 142 סוגי נתונים ייחודיים × 75 סוגי נתונים גנריים
```

#### 8 התחומים שנבדקו

| תחום | כלים לדוגמה | סוגי נתונים רגישים |
|------|-------------|-------------------|
| Entertainment | Movie Database, Web, Apple Music | העדפות אישיות |
| Health & Fitness | Fitness Tracking, PregnancyPal, Hospital Booking | נתונים רפואיים, SSN |
| Smart Home | Lighting Control | מידע על הבית |
| Travel | Weather, Travel Booking, Cloud Drive | דרכון, פרטי אשראי |
| Shopping | Amazon | היסטוריית רכישות |
| Work & Productivity | Slack, Email, Calendar, Cloud Drive | מסמכי עבודה |
| Social | SMS, Instagram, Tinder, Cloud Drive | תקשורת אישית |
| Finance | Banking, Tax Management | חשבון בנק, SSN |

#### 4 אפשרויות הרשאה

לכל בקשת נתונים, המשתתפים בחרו מבין:

```
1. ✅ Yes, always share    — תמיד לשתף (הרשאה קבועה)
2. ✅ Yes, but ask next    — לשתף הפעם, לשאול שוב בפעם הבאה
3. ❌ No, but ask next     — לא לשתף הפעם, לשאול שוב בפעם הבאה
4. ❌ No, never share      — לעולם לא לשתף (חסימה קבועה)
```

(אלו אותן אפשרויות שמציע OpenAI ב-Custom GPTs, למעט "never share")

#### מאפיינים ייחודיים של המחקר

- **Priming לפרטיות** — לפני השאלות, המשתתפים נשאלו על מודעות הפרטיות שלהם, כדי שההחלטות ישקפו את ההתנהגות האמיתית
- **תרחישי טעויות** — ב-25% מהשאלות הוצגו סוגי נתונים **מיותרים** (unnecessary) כדי לבדוק אם המשתתפים מזהים טעויות של הסוכן
- **205 משתתפים** מגוסס דרך Prolific, עם שיעור אישור של 90%

## 4. ממצאים מרכזיים — מה למדנו על העדפות המשתמשים?

### 4.1 ממצא 1: כשהסוכן טועה — פחות הרשאות

```
┌─────────────────────────────────────────────────────────────┐
│         השפעת טעויות הסוכן על החלטות הרשאה                 │
│                                                             │
│  ◆ נתונים נכונים בלבד (Ground Truth = necessary):          │
│    "Always share" = 32.0% מעולם לא בחרו never               │
│    "Never share"  = מינימלי                                 │
│                                                             │
│  ◆ נתונים נכונים + מיותרים (כולל unnecessary):             │
│    "Always share" = 25.6% ← ירידה! ↓                       │
│    "Never share"  = 21.2% ← עלייה! ↑                      │
│                                                             │
│  → משתמשים מזהים טעויות ומגיבים בזהירות                    │
│  → ביצועי הסוכן משפיעים על מתן הרשאות                     │
│  → רלוונטי לתזה: ביצועי סוכן = גורם סיכון!                │
└─────────────────────────────────────────────────────────────┘
```

**תובנה קריטית:** 95.1% מהמשתמשים נותנים הרשאת "always share" לפחות פעם אחת, אבל כשהסוכן טועה — המוכנות לתת הרשאות אוטומטיות **יורדת**. זה מראה ש**ביצועי הסוכן הם גורם אמון קריטי**.

### 4.2 ממצא 2: Over-permissioning שולט

```
תפוצת שיעורי ההרשאות לכל משתתף:

  Over-permission:   90% מהמשתתפים עוברים את 15.4% over-permissioning
  Under-permission:  90% מהמשתתפים עוברים את 8.9%
  Appropriate:       רק 18.7% מהמשתמשים נותנים הרשאה מתאימה ב-90%+ מהמקרים

  → Over-permissioning נפוץ הרבה יותר מ-under-permissioning!
  → אף משתתף לא נתן הרשאות מתאימות ב-100% מהמקרים
```

**מטאפורה:** זה כמו לתת לעוזר חדש את כל המפתחות למשרד "ליתר ביטחון" — רוב האנשים מעדיפים לתת יותר מדי מאשר פחות מדי, גם אם זה מסוכן.

### 4.3 ממצא 3: ההקשר (Domain) משפיע מאוד

**טבלה 1: העדפות הרשאה לפי תחום**

| תחום | Always Share | Never Share | תחום מדאיג — Always | תחום מדאיג — Never |
|------|:-----------:|:-----------:|:------------------:|:-----------------:|
| Entertainment | 55.6% | 2.9% | 60.5% | 2.6% |
| Health & Fitness | 48.9% | 7.7% | 44.6% | 7.6% |
| Smart Home | 41.6% | 9.3% | 43.5% | 7.6% |
| Travel | 36.0% | 10.5% | 20.3% | 3.8% |
| Shopping | 32.2% | 12.0% | 24.0% | 9.3% |
| Work & Productivity | 30.8% | 8.6% | 27.6% | 7.9% |
| Social | 30.4% | 8.1% | 33.6% | 9.9% |
| **Finance** | **22.2%** | **16.9%** | **22.4%** | **13.9%** |

**תובנות:**
- **Entertainment** — הכי מתירני (55.6% always share) — נתונים בעלי רגישות נמוכה
- **Finance** — הכי מגביל (22.2% always share, 16.9% never share) — נתונים רגישים מאוד
- **Travel ו-Shopping** — ירידה חדה כשהמשתתף מסמן את התחום כ"מדאיג מבחינת פרטיות"

### 4.4 ממצא 4: שונות פנים-תחומית בין כלים

**טבלה 2: העדפות לפי כלי ספציפי (דגימה)**

| תחום | כלי | Always | Yes(Once) | No(Once) | Never |
|------|------|:------:|:---------:|:--------:|:-----:|
| Entertainment | Movie Database | 58.3% | 33.7% | 5.9% | 2.1% |
| Entertainment | Web | 34.4% | 52.5% | 4.9% | 8.2% |
| Travel | Cloud Drive | 17.8% | 56.3% | 9.6% | 16.4% |
| Finance | Banking | 24.8% | 50.6% | 11.3% | 13.2% |
| Finance | Tax Management | 22.6% | 50.2% | 9.2% | 18.0% |

**תובנה:** גם בתוך אותו תחום, יש שונות גדולה. Cloud Drive ב-Travel מקבל הרבה פחות "always share" מ-Weather, כי הוא מכיל מסמכי נסיעה רגישים (דרכון, ויזות).

### 4.5 ממצא 5: סוגי נתונים רגישים נחסמים באופן מובהק

```
סוגי נתונים שנחסמו בתדירות הגבוהה ביותר (under-permissioned):

  Child Name          → לא שותף ב-65.1% מהמקרים
  Work Information    → לא שותף ב-57.4%
  Passport Information → לא שותף ב-54.0%
  SSN                 → under-permission ratio של 46.3%
  Account Password    → 50% מאלה שלא שיתפו ציינו Finance כמדאיג

סוגי נתונים שכמעט תמיד שותפו:
  Meeting Details     → appropriate permission של 98.4%
  Fitness Goal        → שונות נמוכה (SD = 0.244)
  Hobbies & Interests → שונות נמוכה (SD = 0.178)
```

### 4.6 ממצא 6: עקביות פנים-אישית, דמיון בין-אישי

```
עקביות בתוך משתמש (intra-user consistency):
  ┌──────────────────────────────────────────────┐
  │  84.5-85.6% מהמשתמשים בתחומי Entertainment  │
  │  ו-Smart Home — העדפה אחידה לכל הנתונים     │
  │  בתחום (SD = 0)                             │
  │                                              │
  │  32.1-44.4% בתחומי Finance, Travel, Shopping │
  │  — עקביות חלקית (SD > 0)                    │
  └──────────────────────────────────────────────┘

דמיון בין משתמשים (inter-user similarity):
  ┌──────────────────────────────────────────────┐
  │  69.7% מזוגות המשתמשים מדווחים על העדפות    │
  │  דומות ב-60%+ מסוגי הנתונים                │
  │  (Jaccard similarity > 0.6)                  │
  │                                              │
  │  → ניתן ללמוד ממשתמשים דומים!               │
  │  → Collaborative Filtering הגיוני!           │
  └──────────────────────────────────────────────┘
```

### 4.7 ממצא 7: דמוגרפיה ומודעות פרטיות משפיעים

```
גורמים שנמצאו משמעותיים:

  גיל: צעירים (<25) → 35.0% always share
       מבוגרים (>55) → 25.5% always share
       → הבדלים דוריים בגישה לפרטיות

  השכלה: רמה גבוהה יותר → פחות הרשאות "always share"

  היכרות עם AI: שימוש תכוף + היכרות → יותר permissive
                 → נוחות עם הטכנולוגיה מגבירה אמון

  מודעות פרטיות: רמה 4 (גבוהה) → הרבה יותר "never share"
                  רמה 2-3 (בינונית) → דווקא יותר permissive מגבוהה
```

## 5. מערכת חיזוי ההרשאות — הארכיטקטורה הטכנית

### 5.1 סקירה כללית

המאמר מציע מודל **היברידי** שמשלב שתי גישות:

```
┌─────────────────────────────────────────────────────────────────┐
│              Hybrid Permission Prediction Model                 │
│                                                                 │
│  ┌──────────────────────┐    ┌────────────────────────────┐    │
│  │  In-Context Learning │    │  Collaborative Filtering   │    │
│  │  (ICL)               │    │  (CF)                      │    │
│  │                      │    │                            │    │
│  │  • למידה מהיסטוריה   │    │  • למידה ממשתמשים דומים    │    │
│  │    אישית של המשתמש   │    │  • LightGCN graph network  │    │
│  │  • OpenAI o3-mini    │    │  • מלא חורים בנתונים       │    │
│  │  • few-shot prompting│    │    דלילים                   │    │
│  │  • מותאם אישית       │    │  • מזהה דפוסים קבוצתיים   │    │
│  └──────────┬───────────┘    └─────────────┬──────────────┘    │
│             │                              │                    │
│             └──────────┬───────────────────┘                    │
│                        │                                        │
│                        ▼                                        │
│             ┌──────────────────────┐                            │
│             │   Hybrid Decision    │                            │
│             │                      │                            │
│             │  ICL prediction      │                            │
│             │  + CF recommendation │                            │
│             │  (as context)        │                            │
│             │  → Final: Allow/Deny │                            │
│             │  + Confidence score  │                            │
│             └──────────────────────┘                            │
│                                                                 │
│  Accuracy: 85.1% overall | 94.4% high-confidence               │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 רכיב 1: In-Context Learning (ICL)

**הרעיון:** להשתמש ב-LLM (OpenAI o3-mini) עם prompting שכולל את כל הפרטים הרלוונטיים.

#### מה נכנס ל-prompt:

```
Role: "permission assistant"

Demographics:
  "a male in the 45-54 age group with a bachelor's degree"

Self-reported:
  AI familiarity: 3/4
  Usage frequency: 2/4
  Privacy consciousness: 4/4
  Trust in AI: 2/4

Permission History (if available):
  Query: "Book a flight to NYC" | Tool: Travel Booking
  Data Type: Credit Card | Decision: Allow

  Query: "Check my health records" | Tool: Hospital Booking
  Data Type: SSN | Decision: Deny
  ...

Current Request:
  Query: "Can you retrieve my tax filing details from last year?"
  Tool: Tax Management
  Data Type: SSN
  → Predict: Allow or Deny? (with confidence score)
```

**למה ICL:** שתי סיבות מרכזיות:
1. יש מעט היסטוריה לכל משתמש — LLM מצליח עם few-shot (1-4 דוגמאות)
2. סוגי נתונים חדשים צצים כל הזמן — LLM מכליל ללא re-training

### 5.3 רכיב 2: Collaborative Filtering (CF)

**הרעיון:** ללמוד מהעדפות של משתמשים דומים — כמו מערכת המלצות.

```
מטריצת ההרשאות (Adjacency Matrix):

              Credit Card  SSN  Email  Name  Health Data  ...
User 1:       Allow        Deny Allow  Allow Deny
User 2:       Allow        Deny Allow  Allow ?           ← חסר!
User 3:       ?            Deny ?      Allow Deny        ← חסר!
User 4:       Deny         Deny Allow  Allow Allow
...

→ LightGCN ממלא את החורים (?) על בסיס דמיון בין משתמשים
```

#### LightGCN — הארכיטקטורה

```
LightGCN (Light Graph Convolution Network):

Nodes = משתמשים + בקשות הרשאה (permission requests)
Edges = הרשאות שניתנו (allow) או נדחו (deny)

┌──────┐          ┌───────────────┐
│User 1│──allow──→│Credit Card    │
│      │──deny───→│for Travel     │
└──────┘          └───────────────┘
     │                    │
     │ similar users      │ similar requests
     │                    │
┌──────┐          ┌───────────────┐
│User 2│──???────→│Credit Card    │
│      │          │for Finance    │
└──────┘          └───────────────┘

→ Embedding iteratively averaged from neighbors
→ Prediction = dot product between user & request embeddings
→ Score > threshold → predicted "allow"
```

**הסף (threshold) נקבע** כך שה-FPR (שיעור שגיאות חיוביות כוזבות) שווה ל-FNR (שיעור שגיאות שליליות כוזבות) — באמצעות 5-fold cross-validation.

**רק predictions עם confidence גבוה נכללים:** FPR < 5% ו-FNR < 5%, שמכסים 35.0% מבקשות ההרשאה.

### 5.4 רכיב 3: המודל ההיברידי

```
שלב 1: CF מייצר recommendation עם confidence score
         ↓
שלב 2: אם confidence גבוה (FPR < 5%, FNR < 5%):
         → הוסף את ה-recommendation כהקשר ל-ICL prompt

         Example added to prompt:
         "Query: Can you retrieve my tax filing details?
          Tool: Tax Management
          Data Type: SSN
          Decision: Deny"    ← recommendation מה-CF

שלב 3: ICL (o3-mini) מקבל החלטה סופית
         → עם/בלי ה-CF recommendation
         → פלט: Allow/Deny + confidence score
```

**למה היברידי:**
- CF לבד: 83.3% accuracy, אבל **לא יכול להתמודד עם סוגי נתונים חדשים**
- ICL לבד: 84.4% accuracy, אבל **מוגבל להיסטוריה אישית בלבד**
- **היברידי: 85.1%** — CF ממלא חורים בנתונים, ICL מכליל לתרחישים חדשים

## 10. רלוונטיות לתזה — ניקוד סיכון דינמי 1-10 לגישת סוכנים ל-MCP

### 10.1 חיבור ישיר — Permission Prediction כחלק מ-Risk Scoring

המאמר הזה מספק **את הצד של המשתמש** במשוואת הסיכון. בעוד שהתזה שלך מתמקדת בהערכת **הסוכן** (כמה הוא מסוכן?), המאמר הזה עונה על השאלה: **"מה המשתמש היה רוצה?"** — וזה חלק קריטי בחישוב ציון הסיכון.

```
התזה שלך (MCP-RSS):
  risk_score = f(agent_behavior, tool_sensitivity, user_preferences, context)
                    ↑                  ↑                  ↑              ↑
                    |                  |                  |              |
              MCPShield           סוג הנתון         המאמר הזה!      המאמר הזה!
              (מאמר 02)          (Table 4)         (user study)    (domain impact)
```

### 10.2 תובנות שניתן לשלב ישירות

#### תובנה 1: ביצועי הסוכן = משקל בסיכון

```
המאמר מראה: סוכן שטועה → פחות אמון → פחות הרשאות

בתזה שלך:
  if agent_error_rate > threshold:
      risk_modifier *= 1.3   ← מגביר את ציון הסיכון

  # סוכן שמבקש נתונים מיותרים = red flag
  unnecessary_data_ratio = unnecessary_requests / total_requests
  risk_score += unnecessary_data_ratio * weight
```

#### תובנה 2: רגישות הנתון כמשקל סיכון

```
מתוך Table 4 + Section 5.3.2:

data_sensitivity_weights = {
    # High variance (hard to predict) = high risk
    "Account Credentials":    0.498,  # SD
    "Employment Details":     0.498,
    "Driver License Number":  0.489,
    "SSN":                    0.475,
    "Bank Account Details":   0.478,

    # Low variance (easy to predict) = low risk
    "Music Listening History": 0.125,
    "Hobbies and Interests":   0.178,
    "Fitness Goal":            0.244,
    "Travel Preferences":     0.246,
}

# בתזה שלך:
risk_score += data_sensitivity_weights[data_type] * access_weight
```

#### תובנה 3: Domain כפקטור הקשרי

```
מתוך Table 1:

domain_risk_multiplier = {
    "Finance":          1.5,   # הכי רגיש (22.2% always share)
    "Shopping":         1.3,
    "Travel":           1.2,
    "Work":             1.1,
    "Social":           1.1,
    "Smart Home":       1.0,
    "Health & Fitness":  0.9,
    "Entertainment":    0.7,   # הכי פחות רגיש (55.6% always share)
}

# בתזה שלך:
risk_score *= domain_risk_multiplier[request_domain]
```

#### תובנה 4: היסטוריה = ירידה בסיכון

```
מתוך Table 6:

# ככל שיש יותר היסטוריה → ביטחון גבוה יותר → סיכון נמוך יותר
confidence_by_history = {
    0:    0.669,  # accuracy with 0 history

### 10.3 מה חסר — הפער שהתזה ממלאת

```
┌────────────────────────────────┬──────────────────────────────────┐
│    המאמר הזה עושה              │    התזה שלך מוסיפה               │
├────────────────────────────────┼──────────────────────────────────┤
│ מחזה: allow / deny (בינארי)   │ מדרג: 1-10 (רציף)              │

│ אכיפה: out of scope           │ אכיפה: חלק מרכזי               │
│ Confidence threshold: binary  │ Risk score: gradual (1-10)      │
└────────────────────────────────┴──────────────────────────────────┘
```

### 10.4 סינתזה — איך לשלב

```
מערכת MCP-RSS משולבת:

שכבה 1: User Preference Layer (מהמאמר הזה)
  → מה המשתמש רוצה? CF + ICL → allow/deny + confidence

שכבה 2: Agent Risk Layer (מהתזה שלך)
  → כמה הסוכן מסוכן? behavioral analysis → risk 1-10

שכבה 3: Combined Decision
  → if user_preference = "always allow" AND agent_risk < 3:
        → auto-approve (no interruption)
  → if user_preference = "never share" OR agent_risk > 7:
        → auto-deny (protect user)
  → else:
        → ask user with context:
          "הסוכן מבקש גישה ל-SSN דרך Tax Management.
           ציון סיכון: 5/10. משתמשים דומים לך חוסמים ב-46% מהמקרים.
           לאשר?"
```

### 10.5 מה הכי חשוב מהמאמר הזה לתזה

```
1. ✅ Data sensitivity weights — משקלי רגישות לסוגי נתונים (Table 4)
   → ישירות לפורמולת הסיכון שלך

2. ✅ Over-permissioning bias — רוב המשתמשים נותנים יותר מדי
   → הצדקה למערכת סיכון שמגנה על המשתמש "מעצמו"

3. ✅ Agent performance matters — ביצועי הסוכן = גורם אמון
   → תמיכה ישירה ברעיון שלך: לדרג סוכנים לפי ביצועים

4. ✅ Cold start solvable — 66.9% בלי היסטוריה, 77.7% עם 1-4
   → מערכת risk scoring יכולה לעבוד מהיום הראשון

5. ✅ Collaborative filtering works — משתמשים דומים = חיזוי טוב
   → ניתן לבנות "community risk profiles" בתזה שלך

6. ✅ Context (domain) is critical — תחום = גורם מפתח
   → risk score חייב לכלול את ההקשר, לא רק את סוג הנתון
