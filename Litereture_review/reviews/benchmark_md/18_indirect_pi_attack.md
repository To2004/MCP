# Indirect PI Attack Dataset

## Source
- **Paper:** Exploiting Web Search Tools of AI Agents for Data Exfiltration (Rall et al., 2025)
- **Link:** https://anonymous.4open.science/r/web-search-exploit-paper-FFC6
- **Year:** 2025

## Format & Size
- **Total samples:** 1,068 unique attack instances (89 templates x 12 variations)
- **Format:** Template-based attack instances with systematic variation across 12 obfuscation types, tested against 28 models
- **Availability:** Anonymous repository (https://anonymous.4open.science/r/web-search-exploit-paper-FFC6) — likely under review, may move to a permanent repository post-publication

## Data Structure

| Field/Column | Data Type | Example Values | Usable As-Is? | Notes |
|---|---|---|---|---|
| Attack template | Text (89 templates) | Base prompt injection payloads targeting web search tools for data exfiltration | Yes | Core attack patterns before variation is applied |
| Variation type | Categorical (12 types) | LLM-based fuzzing, Base64 encoding, Unicode obfuscation, random capitalization, translation, etc. | Yes | Generated using PyRIT converters — systematic and reproducible |
| Attack instance | Text | Concrete instantiation of template + variation | Yes | 1,068 total = 89 x 12 |
| Target model | Categorical (28 models) | Models from X-AI, Inception, Qwen, Meta, Anthropic, OpenAI, Google, DeepSeek, Mistral | Yes | Broad provider coverage |
| Vulnerability result | Boolean/float | Success/failure per model per attack | Yes | Per-model vulnerability rates available |
| Model vulnerability rate | Float (%) | X-AI grok-4: 72.4%, OpenAI/Google/Amazon: near 0% | Yes | Aggregate per-model scores — ready to use |
| Attack focus | Categorical | Data exfiltration via web search tools | Yes | Narrow but deep — single attack vector studied thoroughly |

## Proposed Risk Dimensions

### Evasion Technique Sophistication
- **Feeding columns:** Variation type (12 types)
- **Proposed scale:** 1-10 where basic text variations = 2-3, encoding-based obfuscation = 5-7, and advanced evasion (LLM-based fuzzing, multi-technique combination) = 8-10
- **Derivation:** Rank the 12 variation types by detection difficulty. Proposed ordering:
  - **Low evasion (2-3):** Random capitalization, simple rephrasing — surface-level changes easily caught by normalization
  - **Medium evasion (4-6):** Translation (language change defeats keyword matching), Unicode obfuscation (visually similar but different codepoints)
  - **High evasion (7-8):** Base64 encoding (payload hidden from text-based scanners), multi-step obfuscation chains
  - **Very high evasion (9-10):** LLM-based fuzzing (adversarially optimized, adapts to defenses)

  At runtime, score an input by detecting which obfuscation techniques are present. Multiple techniques stacked together increase the score (e.g., Base64 + Unicode + translation = 9). The 12 variation types generated by PyRIT converters provide a systematic coverage map — a defense that catches all 12 is robust; one that misses 4+ has gaps.

### Model-Specific Vulnerability Adjustment
- **Feeding columns:** Model vulnerability rate per target model
- **Proposed scale:** 1-10 directly mapped from the model's measured vulnerability rate
- **Derivation:** The per-model vulnerability data provides a direct risk multiplier. X-AI grok-4 at 72.4% vulnerability maps to a risk score of 7-8 for any tool invocation processed by that model. OpenAI/Google/Amazon at near 0% map to 1-2. For an MCP deployment, if you know which LLM backend is processing tool calls, you can look up its vulnerability rate from this benchmark and apply it as a base risk modifier. Formula: `model_risk = ceil(vulnerability_rate / 10)`, so 72.4% → 8, 15% → 2, 0% → 1. This is the only benchmark that provides per-model vulnerability rates across 28 models from 9 providers.

### Exfiltration Vector Risk
- **Feeding columns:** Attack focus (data exfiltration via web search), attack template content
- **Proposed scale:** 1-10 where the score reflects the severity of the exfiltration channel
- **Derivation:** All 1,068 instances target the same vector — data exfiltration through web search tool abuse. The risk score for this dimension is fixed at 7-8 for any tool invocation that involves web search or external HTTP requests, because the paper demonstrates that this channel is reliably exploitable (72.4% success rate on the most vulnerable model). Within this vector, score higher if the exfiltration target contains sensitive data (PII, credentials, internal documents = 9-10) vs. non-sensitive data (public queries = 4-5). The narrow focus is a limitation for general risk scoring but provides depth on this specific vector.

## Data Quality Notes
- The 89 x 12 structure is clean and systematic — every template gets every variation, so coverage is uniform. No missing combinations or sparse cells.
- The 12 variation types are generated using PyRIT converters, which is a well-documented attack generation framework. This means the variations are reproducible and extensible — you can add new converter types to generate additional variations from the same 89 templates.
- The 28-model evaluation is exceptionally broad. Most benchmarks test 3-5 models. Having 28 models across 9 providers (X-AI, Inception, Qwen, Meta, Anthropic, OpenAI, Google, DeepSeek, Mistral) gives the best available cross-model comparison data.
- The vulnerability gap is enormous: 72.4% (grok-4) vs. near 0% (OpenAI/Google/Amazon). This suggests model architecture and safety training matter far more than the specific attack variation, which is an important finding for risk scoring — the model-specific adjustment may dominate other risk dimensions.
- The dataset focuses exclusively on data exfiltration via web search. It does not cover other attack vectors (direct harm, privilege escalation, etc.), so it is a deep but narrow contribution.
- The anonymous repository URL suggests the paper is under review. The data may move or change post-publication.

## Usefulness Verdict
This dataset's unique contribution is the model-specific vulnerability dimension. No other benchmark in the survey tests 28 models and provides per-model vulnerability rates, making this the only source for calibrating a "which LLM backend is processing this request" risk factor. For enterprise MCP deployments where the backend model is known, this data directly feeds a risk adjustment: an agent powered by grok-4 processing web search results should score 7-8 on model vulnerability, while an agent powered by GPT-4 or Gemini should score 1-2.

The 12 obfuscation variation types also provide a practical input-screening checklist. If a risk scorer detects Base64 encoding, Unicode obfuscation, or signs of LLM-based fuzzing in a tool input, it can immediately elevate the risk score for that request. The systematic template x variation structure makes it straightforward to build a training set for an obfuscation detector. The main limitation is the narrow attack vector — everything is about web search exfiltration. For a multi-dimensional risk scorer, this benchmark feeds two dimensions well (model vulnerability, evasion technique) but is silent on others (harm type, permission scope, tool trustworthiness).
