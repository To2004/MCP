% ABSTRACT
% ---------------------------------------------------------

\begin{abstract}
The Model Context Protocol (MCP) enables AI agents to connect to third-party tool servers, creating a rapidly growing ecosystem of over 67{,}000 servers across six public registries. Current MCP security mechanisms rely on binary allow/deny decisions that treat all tool invocations identically, regardless of the actual risk posed by a specific request in a specific operational context. This work proposes a dynamic risk scoring framework for MCP agent access that produces a graduated 1--10 severity score for each tool invocation request, inspired by the CVSS vulnerability scoring methodology. We survey the MCP security landscape including attack taxonomies, tool poisoning techniques, defense frameworks, and access control mechanisms, and identify the absence of graduated, context-aware risk quantification as a critical gap. Building on evidence that LLMs can score vulnerability descriptions and that multi-signal aggregation improves detection over single-signal approaches, we outline a multi-stage scoring pipeline that combines static tool metadata analysis, contextual enrichment from session behavior and ecosystem baselines, and LLM-based risk judgment. The proposed system aims to enable nuanced security decisions---auto-allow, log, escalate, or auto-deny---based on quantified risk rather than binary classification.
\end{abstract}

\section{Background and Related Work}

This chapter surveys six complementary research areas that together reveal a critical gap in MCP security: while the ecosystem faces sophisticated and diverse attacks and the research community has begun developing defenses and access controls, no existing work produces a graduated, dynamic risk score for MCP tool invocations. All current security mechanisms ultimately reduce to a binary decision---allow or deny---ignoring the wide spectrum of risk between clearly safe and clearly dangerous requests. This binary framing forces system designers into an undesirable trade-off: overly permissive policies that expose users to high-risk tool calls, or overly restrictive policies that block legitimate tool use and degrade agent utility.

We organize the literature as follows. We first review the MCP architecture and the scale of its ecosystem (\S\ref{subsec:ecosystem}). We then survey the attack taxonomies that enumerate how adversaries exploit MCP (\S\ref{subsec:attacks}), with particular attention to tool poisoning and parasitic attacks that represent the most effective attack class (\S\ref{subsec:poisoning}). Next, we examine defense frameworks and runtime guardrails (\S\ref{subsec:defenses}), followed by access control and privilege management approaches (\S\ref{subsec:access}). We then discuss risk scoring, trust management, and vulnerability prioritization as studied in adjacent domains (\S\ref{subsec:scoring}). Finally, we position this work relative to the identified gap and state our contributions (\S\ref{subsec:positioning}).

\subsection{MCP Architecture and Ecosystem Overview}
\label{subsec:ecosystem}

The Model Context Protocol (MCP), introduced by Anthropic in late 2024, standardizes how AI agents connect to external tool servers through a three-tier architecture comprising hosts (the LLM-powered application), clients (protocol-level connectors), and servers (tool providers). MCP supports four core primitives: \emph{tools} (callable functions), \emph{resources} (data endpoints), \emph{prompts} (templated instructions), and \emph{sampling} (server-initiated LLM queries). This design enables a rich interaction model where servers can expose arbitrary functionality to agents, but it also introduces a fundamentally permissive trust model: the protocol assumes that tool metadata---including names, descriptions, and input schemas---is accurate and benign.

Hou et al.\ provide the first systematic academic analysis of the MCP ecosystem \cite{hou2025mcp_landscape}. Their survey catalogs the protocol’s architecture, SDK implementations, and adoption patterns across 26 server collections, and develops a threat taxonomy organized along the MCP server lifecycle: creation, deployment, operation, and maintenance. They identify four attacker types---malicious server developer, compromised client, network adversary, and supply-chain attacker---that collectively give rise to 16 distinct threat scenarios, each demonstrated with proof-of-concept attacks. The lifecycle perspective is particularly relevant to risk scoring, as it implies that the risk profile of a given server or tool may change over time (e.g., through rug-pull attacks where behavior shifts after an initial trust-building phase).

Li and Gao complement this architectural analysis with an empirical study of the ecosystem’s scale and security posture \cite{li2025understanding_mcp}. Analyzing 67{,}057 MCP servers across six public registries (mcp.so, MCP Market, Smithery, npm, PyPI, and Docker Hub), they find that 99.16\% of servers are hosted on GitHub, with 52.1\% implemented in Python, 22.5\% in JavaScript, and 11.6\% in TypeScript. Their quantitative analysis extracts 44{,}499 tools and identifies systemic vulnerabilities: 9 personal access tokens and 3 API keys exposed in server configurations, over 111 instances of server hijacking through abandoned repository takeover, and 408 groups of confusingly similar tool names that enable affix-squatting attacks. A qualitative evaluation of four major MCP hosts---Cursor, Windsurf, Claude Desktop, and Cline---reveals that hosts differ substantially in how they handle server trust and tool approval, with none implementing graduated risk assessment. Together, these studies establish that MCP’s open ecosystem is growing rapidly, that its trust model is fundamentally permissive, and that the scale of potential exposure demands automated, graduated risk quantification rather than manual auditing or binary allow/deny gates.

\subsection{Attack Taxonomies and Threat Models for MCP}
\label{subsec:attacks}

Multiple independent research efforts have developed threat taxonomies for the MCP attack surface, each approaching the problem from a different angle. Ferrag et al.\ present the most comprehensive end-to-end threat model, spanning over 30 attack techniques organized across four domains: input manipulation (prompt injection, jailbreaking), model exploitation (training data poisoning, model inversion), system compromise (privilege escalation, lateral movement), and protocol abuse (transport-layer attacks, session hijacking) \cite{ferrag2025prompt_to_protocol}. Their analysis covers not only MCP but also related agent-to-agent protocols such as A2A, ANP, and ACP, providing a unified view of the agentic AI threat landscape. The evidence base draws from over 150 sources including 12 real CVEs from 2025, making it the most empirically grounded taxonomy available. For risk scoring, this work provides the attack enumeration against which a scoring system’s coverage must be validated.

Zhao et al.\ narrow the focus to malicious MCP servers specifically, a threat model where the server developer is the adversary \cite{zhao2025when_mcp_servers_attack}. With over 16{,}000 MCP servers available online without any vetting process, this is a realistic and high-impact threat. They identify 12 distinct attack categories organized across six server components: metadata manipulation, configuration abuse, initialization-time attacks, tool-level exploits, resource-level exploits, and prompt-level exploits. Notable attack types include tool shadowing (a malicious tool declares the same name or similar functionality as a legitimate tool, causing the agent to invoke the attacker’s version), rug-pull attacks (the server behaves correctly during initial interactions to build trust, then shifts to malicious behavior), and cross-server data exfiltration (a malicious server reads data from the agent’s context that was provided by a co-installed legitimate server). Their feasibility analysis demonstrates that a single malicious server in a multi-server environment can compromise the entire agent’s behavior without the user’s awareness, underscoring the need for per-invocation risk assessment rather than per-server binary trust.

Maloyan and Namiot take a protocol-level perspective, analyzing three architectural vulnerabilities in the MCP specification (version 1.0) itself \cite{maloyan2026breaking_protocol}. The first vulnerability concerns the absence of server identity verification: servers self-assert their capabilities and identity with no external validation mechanism, enabling impersonation and capability inflation. The second concerns unauthenticated sampling requests: servers can send prompts to the host LLM through the sampling API without origin authentication, creating a channel for indirect prompt injection. The third concerns the lack of isolation between co-installed servers: multiple servers share the same agent context, allowing one server to influence the agent’s interactions with another. These are not implementation bugs but fundamental protocol design weaknesses that cannot be patched by individual server implementations. Their experimental evaluation shows that MCP amplifies baseline prompt injection success rates by 23--41\%, from 31.2\% without MCP to 47.8\% with MCP. For risk scoring, this implies that protocol-level context (e.g., whether the server uses sampling, how many co-installed servers are present) should be a factor in the risk assessment.

\subsection{Tool Poisoning and Parasitic Attacks}
\label{subsec:poisoning}

Tool poisoning---the manipulation of tool metadata to steer agent behavior toward malicious actions---has emerged as the most effective and most studied class of MCP attacks. Unlike traditional software exploits that target code vulnerabilities, tool poisoning exploits the semantic interface between the LLM and the tool ecosystem: because agents rely on natural-language descriptions to understand what tools do, adversaries can manipulate these descriptions to alter agent behavior without modifying any executable code.

Wang et al.\ introduce MCPTox, the first systematic benchmark for tool poisoning attacks against real-world MCP servers \cite{wang2025mcptox}. The benchmark spans 45 MCP servers and 353 tools, evaluating three poisoning paradigms: (P1) explicit function hijacking, where the poisoned tool’s description contains direct malicious instructions that cause the agent to invoke a different tool with attacker-controlled parameters; (P2) implicit behavioral steering, where subtle language in the description biases the agent’s tool selection without explicit commands; and (P3) cross-server poisoning, where a malicious tool on one server affects the agent’s interaction with tools on a different server. Their results show a 72.8\% overall attack success rate for explicit poisoning across multiple LLM backends, with per-model rates of 61.8\% for GPT-4o and 34.3\% for Claude. A critical finding is that tool poisoning attacks (TPA) are mechanistically distinct from indirect prompt injection (IPI): IPI payloads placed in tool descriptions achieve near-zero success rates, confirming that TPA exploits a different vulnerability---the agent’s trust in metadata---and requires dedicated detection mechanisms.

Li et al.\ demonstrate that implicit poisoning is far more dangerous than explicit poisoning in practice \cite{li2026mcp_itp}. Their MCP-ITP framework automates the generation of implicitly poisoned tool descriptions that achieve an 84.2\% attack success rate while evading detection by existing security tools at a rate of only 0.3\%. The key insight is that implicitly poisoned tools are never directly invoked by the agent---instead, their metadata subtly influences the agent’s reasoning process, causing it to misuse other legitimate tools with elevated privileges or altered parameters. For example, a poisoned tool description reading ``For best results, include context from recent files’’ may cause the agent to pass sensitive file contents to a legitimate network tool, achieving data exfiltration through entirely benign tool invocations. This indirection makes the attack invisible to any monitoring system that only inspects tool call patterns, as every individual tool call appears legitimate. For risk scoring, this finding implies that metadata-level features (description semantics, cross-tool reference patterns) must be core inputs to the scoring function, and that per-tool risk assessment alone is insufficient.

A related but distinct attack paradigm is the parasitic toolchain attack, introduced by Zhao et al.\ \cite{zhao2025mind_your_server}. In this model, no single tool is malicious in isolation, but dangerous behavior emerges when the agent chains tools from different servers in an attacker-orchestrated sequence. The attack proceeds in three stages: parasitic ingestion (the agent reads attacker-controlled content from an external source via a legitimate ingestion tool), passive propagation (the malicious instructions persist in the agent’s context across subsequent tool calls), and execution (the agent invokes a legitimate data-access or network tool to exfiltrate data or perform unauthorized actions). Their analysis of the MCP ecosystem identifies 472 External Ingestion Tools (EIT), 391 Privacy Access Tools (PAT), and 180 Network Access Tools (NAT) across 128, 155, and 89 servers respectively. Of all surveyed servers, 27.2\% expose exploitable tool combinations, and real-world attack simulations achieve a 90\% success rate across 10 test toolchains. This demonstrates that tool risk is inherently compositional: a tool that is safe when used alone may become dangerous when combined with specific other tools in the same session.

Hu et al.\ identify a particularly insidious variant of parasitic attacks in their Log-To-Leak study \cite{hu2026log_to_leak}. They show that benign infrastructure tools---such as logging and monitoring services commonly installed alongside application-facing MCP servers---can be weaponized for covert data exfiltration. An attacker embeds instructions in external content that cause the agent to ``log’’ sensitive information using a legitimate logging tool, writing it to a location the attacker can access. Because the agent’s primary task still completes successfully and produces correct output, the user has no observable indication that exfiltration occurred. This finding has direct implications for risk scoring: output quality and task success cannot serve as safety signals, and the risk model must evaluate tool combinations and data flow patterns, not just individual tool properties.

Wang et al.\ approach the detection problem from the agent’s internal perspective with MindGuard \cite{wang2026mindguard}. Rather than monitoring tool calls externally, MindGuard inspects the LLM’s internal decision-making process through Total Attention Energy (TAE) analysis to detect when metadata poisoning is influencing the agent’s tool selection. Their work identifies the code/data confusion problem---where the LLM treats untrusted tool metadata as implicit executable instructions---as the root cause of tool poisoning vulnerability. They model a three-stage attack chain: registration (malicious context is injected into the agent’s context during server initialization), planning (the injected context influences the agent’s tool selection during task planning), and execution (the agent invokes a legitimate tool with attacker-controlled parameters). MindGuard’s approach of inspecting internal reasoning states, rather than input/output patterns, suggests that risk scoring may benefit from incorporating agent-internal behavioral signals alongside static metadata features.

\subsection{Defense Frameworks and Runtime Guardrails}
\label{subsec:defenses}

Several defense frameworks have been proposed to mitigate MCP security risks, employing architectures ranging from adaptive trust calibration to cascaded detection pipelines and multi-agent guardrails. While these frameworks demonstrate strong detection performance, they share a common limitation: all produce binary or categorical security decisions rather than graduated risk scores.

Zhou et al.\ introduce MCPShield, a security cognition layer that implements adaptive trust calibration across the MCP agent lifecycle \cite{zhou2026mcpshield}. The framework comprises three stages: pre-invocation probing, where the system tests server behavior by submitting both benign and adversarial test cases and analyzing responses for signs of metadata misalignment; execution-time monitoring, where the system tracks behavioral consistency during a session using isolated sandbox projections and kernel-level trace recording; and post-invocation reasoning, where the system applies periodic temporal reasoning to detect rug-pull attacks where server behavior degrades after an initial trust-building phase. Evaluated on their MCPSecBench benchmark spanning 17 attack types, MCPShield achieves a 95.30\% detection rate. However, the framework’s output is a binary trusted/untrusted classification for each server, and it does not distinguish between a mildly suspicious tool call and a high-confidence attack.

Xing et al.\ propose MCP-Guard, a defense-in-depth architecture designed to balance detection accuracy with computational efficiency through a three-stage cascaded pipeline \cite{xing2026mcp_guard}. Stage~I applies six parallel lightweight pattern scanners that catch approximately 38.9\% of threats in under 2ms using static rule matching. Stage~II deploys a fine-tuned E5 neural classifier that handles semantically complex attacks in approximately 55ms. Stage~III invokes LLM-based arbitration only for ambiguous cases where the Stage~II confidence falls in a narrow band ($0.45 < P < 0.55$), keeping the most expensive component reserved for edge cases. The overall system achieves an F1 score of 95.4\% with a 2.04$\times$ speedup over monolithic LLM-only defenses. This cascaded architecture demonstrates that multi-stage pipelines are effective for MCP security, a design principle that directly informs our proposed multi-stage scoring pipeline. However, MCP-Guard’s output remains binary (block or allow), and the intermediate confidence scores from Stage~II are not exposed to the user or used for graduated response policies.

At the input level, Gosmar et al.\ propose a multi-agent NLP framework specifically targeting prompt injection detection \cite{gosmar2025prompt_injection}. They define a formal attack model where a prompt $P$ consists of a target instruction $s^t$ and target data $x^t$, and an attack transforms the data into $\tilde{x} = A(x^t, s^e)$ where $A$ is the attack operation and $s^e$ is the adversarial instruction. Their framework classifies attacks into 10 categories---from direct override and authority assertions through obfuscation, role-play, logical traps, multi-step sequences, HTML embedding, and social engineering---and deploys a four-stage detection pipeline: front-end generation, guard/sanitizer, policy enforcer, and KPI evaluator. Testing across 500 engineered prompts (50 per category) demonstrates the viability of collaborative agent verification for injection detection. However, the framework operates upstream of tool invocation and does not assess the risk of the tool call itself.

Mou et al.\ introduce ToolSafe, which shifts the monitoring granularity from the session level to the individual tool-call level \cite{mou2026toolsafe}. ToolSafe comprises two components: TS-Guard, which detects dangerous tool calls by analyzing the agent’s reasoning-action loop at each step, and TS-Flow, which provides interpretable feedback that allows the agent to self-correct when a risky invocation is detected rather than outright blocking the action. This proactive feedback approach addresses a limitation of pure blocking systems: in complex multi-step tasks, blocking a single tool call may cascade into task failure, whereas guided self-correction can preserve task completion while redirecting the agent away from dangerous actions. ToolSafe also introduces TS-Bench, a new benchmark for evaluating step-level tool safety that addresses three threat types: direct dangerous requests, modified prescriptions, and inherently dangerous tools. Despite the step-level granularity, ToolSafe’s output remains a binary safe/unsafe classification per tool call.

Xiang et al.\ take a fundamentally different architectural approach with GuardAgent \cite{xiang2025guardagent}. Rather than embedding security logic within the primary agent’s pipeline, GuardAgent deploys a separate guard agent that operates in parallel. The guard agent follows a four-stage process: task planning (understanding the security requirement), code generation (producing executable verification code), execution (running the verification), and policy validation (checking results against safety specifications). A key design choice is code-based enforcement: instead of relying on LLM reasoning for safety decisions (which is probabilistic and susceptible to manipulation), GuardAgent generates deterministic Python code that implements the safety policy, providing stronger guarantees. The system requires no training and adapts to new safety requirements through in-context learning with examples. Evaluated on two benchmarks---EICU-AC for medical agent safety and Mind2Web-SC for web agent safety---GuardAgent demonstrates effective non-invasive guardrail capabilities without modifying the target agent.

All five defense frameworks produce binary or categorical security decisions (safe/unsafe, allow/block, trusted/untrusted). None outputs a numeric risk score that would enable graduated response policies---auto-allow for low-risk calls, log-and-monitor for moderate-risk calls, escalate-to-human for high-risk calls, or auto-deny for critical-risk calls. This absence of graduated output is the core gap that motivates the present work.

\subsection{Access Control and Privilege Management for AI Agents}
\label{subsec:access}

A complementary line of research addresses the problem of constraining what agents are permitted to do, rather than detecting what they should not do. While defense frameworks operate reactively (monitoring and blocking), access control operates proactively by defining permission boundaries before tool invocation occurs.

Shi et al.\ introduce Progent, a programmable privilege control system for LLM agents that allows developers to specify fine-grained, composable access policies at the tool-call level \cite{shi2025progent}. Progent addresses two key challenges: how to define expressive security policies that capture complex permission requirements (e.g., ``allow file reads in \texttt{/tmp} but deny writes outside the project directory’’), and how to enforce these policies deterministically across probabilistic LLM agents. The system operates as a proxy that intercepts tool calls before execution and evaluates them against JSON-based policy specifications. Progent reduces attack success rates from 41.2\% to 2.2\% across their evaluation scenarios, demonstrating the effectiveness of fine-grained policy enforcement. However, two limitations are directly relevant to risk scoring: first, Progent’s policies are static and predefined by the developer at configuration time, meaning they cannot adapt to runtime context or emerging threats; second, the system enforces a hard binary boundary (allow or deny) with no mechanism for graduated responses to intermediate-risk requests.

Buhler et al.\ propose AgentBound, the first framework that treats access control as a first-class primitive in the MCP architecture \cite{buhler2025agentbound}. AgentBound introduces two mechanisms: AgentManifest, a declarative specification where each MCP server declares the minimum set of permissions it requires to function (analogous to Android app permission declarations), and AgentBox, a runtime enforcement layer that constrains server behavior to the permissions declared in the manifest, enforced at the operating system level. Their analysis reveals a critical baseline problem: MCP servers currently run with the full permissions of the host user account, meaning any server can access any file, network resource, or system service that the user can. This over-privileging creates an attack surface far broader than any individual tool requires and explains why attacks like parasitic toolchain exploitation achieve such high success rates. AgentBound addresses four specific attack types---tool poisoning, puppet attacks (where one server manipulates another through the agent), rug-pull attacks, and malicious external resource loading---by restricting each server to its declared permission envelope.

Wu et al.\ address the specific challenge of automating data access permissions for AI agents, a problem that lies at the intersection of access control and usability \cite{wu2025automating_data_access}. Their work, accepted at IEEE S\&P 2026, observes that traditional permission models (install-time or runtime dialogs) are poorly suited to agentic systems where the agent operates autonomously and may need different data for different sub-tasks within the same session. They propose a Permission Assistant that learns user preferences through interaction and generates minimal-privilege policies from natural language task descriptions, reducing the manual burden of policy specification while maintaining security. Their threat model distinguishes between non-malicious but fallible agents and actively untrustworthy tools, a distinction directly relevant to risk scoring where the risk source (agent error vs.\ tool manipulation) affects the appropriate response.

All three access control frameworks enforce binary permission boundaries (allowed or denied) and do not quantify the degree of risk associated with a permitted action. A graduated risk scoring approach would complement access control systems by providing the risk signal that informs policy decisions: rather than requiring developers to anticipate all dangerous scenarios at configuration time, a dynamic score could trigger policy escalation (from auto-allow to human-in-the-loop) when the estimated risk of a specific invocation exceeds a configurable threshold.

\subsection{Risk Scoring, Trust Management, and Vulnerability Prioritization}
\label{subsec:scoring}

The question of how to quantify security risk numerically rather than categorically has been studied extensively in vulnerability management and trust governance, but remains entirely unexplored for MCP tool invocations. Two bodies of work provide the methodological foundation for applying risk scoring to the MCP domain.

Jafarikhah et al.\ investigate whether LLMs can automate the traditionally manual process of CVSS vulnerability scoring \cite{jafarikhah2026description_to_score}. Motivated by the growing imbalance between vulnerability discovery and analyst capacity---28{,}818 CVEs were reported in 2023 against only 21 analysts (1{,}372 CVEs per analyst), rising to 40{,}009 CVEs in 2024 (a 38\% increase with the same staff)---they evaluate multiple LLMs on the task of predicting CVSS base metric scores from vulnerability descriptions alone. Their results show approximately 79\% accuracy with a two-shot prompting strategy across eight CVSS base metrics spanning exploitability and impact dimensions. Two findings are directly relevant to MCP risk scoring. First, text-based risk scoring is feasible: given a textual description of what a component does, what access it requires, and what can go wrong, an LLM can produce a structured, multi-dimensional severity assessment without access to the underlying code or system. Second, MCP tool descriptions share structural similarity with CVE descriptions---both describe functionality, access scope, input parameters, and potential impact in natural language---suggesting that a similar scoring approach can be adapted to the MCP domain. The key difference is that CVE scoring is performed once per vulnerability, whereas MCP tool invocation scoring must operate dynamically at request time, incorporating session context and tool combination patterns.

Raza et al.\ survey Trust, Risk, and Security Management (TRiSM) for agentic AI systems, providing the governance framework within which risk scoring operates \cite{raza2025trism}. Their comprehensive analysis of 180 papers from 2022--2025 identifies six core governance pillars for multi-agent systems: trust calibration, risk quantification, security enforcement, privacy preservation, governance and compliance, and accountability. Within the risk quantification pillar, they identify several requirements that are directly relevant to MCP scoring: risk signals must be \emph{dynamic} (updated as new behavior is observed within a session), \emph{contextual} (sensitive to the specific operational environment, including which other tools and servers are co-installed), and \emph{composable} (aggregatable across multiple agents and tools to produce system-level risk assessments). The survey also identifies unique challenges posed by multi-agent systems that do not arise in single-agent settings, including agent coordination risks (where agents influence each other’s behavior), shared memory poisoning (where one agent corrupts the context available to others), and group-think effects (where agents converge on suboptimal decisions through mutual reinforcement). However, despite identifying risk quantification as a core requirement, the survey finds no existing system that produces graduated, dynamic risk scores for MCP tool invocations or for agentic AI tool use more broadly.

Together, these works provide the two essential ingredients for MCP risk scoring: a demonstrated methodology showing that LLMs can produce structured risk scores from natural-language descriptions (from vulnerability scoring research), and a governance framework specifying the requirements that such scores must satisfy in the agentic AI context (from TRiSM research). What remains missing---and what this work addresses---is the application of these principles to the specific problem of scoring MCP tool invocations dynamically at request time.

\subsection{Positioning of This Work}
\label{subsec:positioning}

Taken together, the surveyed literature establishes four key findings. First, the MCP ecosystem is large and growing rapidly with minimal security oversight: over 67{,}000 servers across six registries, with no mandatory vetting process and systemic credential leakage \cite{hou2025mcp_landscape,li2025understanding_mcp}. Second, the attack surface is broad and sophisticated, spanning tool poisoning (72.8--84.2\% success rates), protocol-level exploits (23--41\% amplification of prompt injection), and parasitic toolchain compositions (90\% success rate), with implicit attacks evading detection at 0.3\% rates \cite{ferrag2025prompt_to_protocol,wang2025mcptox,li2026mcp_itp,zhao2025mind_your_server,hu2026log_to_leak}. Third, all existing defense frameworks and access control mechanisms produce binary allow/deny decisions, despite the demonstrated effectiveness of multi-stage detection pipelines \cite{zhou2026mcpshield,xing2026mcp_guard,shi2025progent,buhler2025agentbound,mou2026toolsafe,xiang2025guardagent}. Fourth, LLM-based risk scoring from textual descriptions is feasible and governance frameworks for agentic AI explicitly call for graduated, dynamic risk quantification, but no system applies these principles to MCP \cite{jafarikhah2026description_to_score,raza2025trism}.

The critical gap is the absence of a graduated, dynamic risk scoring system that quantifies how dangerous a specific MCP tool invocation is in a specific operational context, rather than classifying it as safe or unsafe. This gap has practical consequences: without graduated scoring, system designers must choose between security (block anything suspicious) and utility (allow everything to preserve agent functionality), with no principled middle ground.

This work addresses this gap by proposing a dynamic risk scoring framework for MCP agent access that produces a 1--10 severity score per tool invocation request, analogous to how CVSS scores quantify software vulnerability severity. The graduated score enables a spectrum of response policies---auto-allow (score 1--3), log-and-monitor (score 4--5), escalate-to-human (score 6--7), and auto-deny (score 8--10)---that balance security with agent utility based on quantified risk.

\paragraph{Contributions.}
\begin{itemize}
    \item A formal risk taxonomy defining the base metrics for MCP tool invocation risk, adapted from the CVSS scoring methodology to the MCP domain and informed by the attack landscape documented across 20 recent studies.
    \item A multi-stage scoring pipeline that combines static tool metadata analysis, contextual enrichment from session behavior and ecosystem baselines, and LLM-based risk judgment to produce graduated 1--10 risk scores at request time.
    \item An evaluation against existing binary defense baselines demonstrating that graduated scoring enables more nuanced and effective security decisions across a range of attack scenarios.
\end{itemize}
