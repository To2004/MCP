% ABSTRACT
% ---------------------------------------------------------

\begin{abstract}
The Model Context Protocol (MCP) enables AI agents to connect to third-party tool servers, creating a rapidly growing ecosystem of over 67{,}000 servers across six public registries. Current MCP security mechanisms rely on binary allow/deny decisions that treat all tool invocations identically, regardless of the actual risk posed by a specific request in a specific operational context. This work proposes a dynamic risk scoring framework for MCP agent access that produces a graduated 1--10 severity score for each tool invocation request, inspired by the CVSS vulnerability scoring methodology. We survey the MCP security landscape including attack taxonomies, tool poisoning techniques, defense frameworks, and access control mechanisms, and identify the absence of graduated, context-aware risk quantification as a critical gap. Building on evidence that LLMs can score vulnerability descriptions and that multi-signal aggregation improves detection over single-signal approaches, we outline a multi-stage scoring pipeline that combines static tool metadata analysis, contextual enrichment from session behavior and ecosystem baselines, and LLM-based risk judgment. The proposed system aims to enable nuanced security decisions---auto-allow, log, escalate, or auto-deny---based on quantified risk rather than binary classification.
\end{abstract}

\section{Background and Related Work}

This chapter surveys six complementary research areas that together reveal a critical gap in MCP security: while the ecosystem faces sophisticated attacks and has begun developing defenses and access controls, no existing work produces a graduated, dynamic risk score for MCP tool invocations. All current security mechanisms reduce to a binary decision---allow or deny---ignoring the spectrum of risk between clearly safe and clearly dangerous requests.

\subsection{MCP Architecture and Ecosystem Overview}

The Model Context Protocol (MCP), introduced by Anthropic in late 2024, standardizes how AI agents connect to external tool servers through a host-client-server architecture. MCP supports four core primitives: \emph{tools} (callable functions), \emph{resources} (data endpoints), \emph{prompts} (templated instructions), and \emph{sampling} (server-initiated LLM queries). The protocol assumes that tool metadata---names, descriptions, and input schemas---is trustworthy, yet servers are contributed by untrusted third parties with no mandatory security review.

Hou et al.\ provide the first systematic analysis of the MCP ecosystem \cite{hou2025mcp_landscape}. They catalog the protocol’s architecture and adoption patterns across 26 server collections, identifying four attacker types (malicious server developer, compromised client, network adversary, and supply-chain attacker) that give rise to 16 distinct threat scenarios with proof-of-concept demonstrations. The lifecycle perspective---creation, deployment, operation, maintenance---implies that the risk profile of a server or tool may change over time, e.g., through rug-pull attacks where behavior shifts after trust is established.

Li and Gao quantify the ecosystem’s scale through an empirical analysis of 67{,}057 MCP servers across six public registries, encompassing 44{,}499 tools \cite{li2025understanding_mcp}. Their study uncovers credential leakage (9 personal access tokens and 3 API keys exposed in server configurations), server hijacking vulnerabilities in over 111 instances, and 408 groups of confusingly similar tool names enabling affix-squatting attacks. A qualitative evaluation of four major MCP hosts (Cursor, Windsurf, Claude Desktop, and Cline) reveals that none implements graduated risk assessment. Together, these studies establish that MCP’s ecosystem is growing rapidly with a fundamentally permissive trust model, demanding automated risk quantification rather than manual auditing.

\subsection{Attack Taxonomies and Threat Models for MCP}

Multiple independent efforts have developed threat taxonomies for the MCP attack surface. Ferrag et al.\ present an end-to-end threat model spanning over 30 attack techniques across four domains---input manipulation, model exploitation, system compromise, and protocol abuse---covering MCP and related protocols (A2A, ANP, ACP) \cite{ferrag2025prompt_to_protocol}. Their evidence-driven analysis of over 150 sources, including 12 real CVEs from 2025, provides the most comprehensive attack enumeration available.

Zhao et al.\ focus on malicious MCP servers, identifying 12 attack categories across six server components \cite{zhao2025when_mcp_servers_attack}. Notable attacks include tool shadowing (overriding legitimate tools), rug-pull attacks (delayed malicious behavior), and cross-server data exfiltration. Their analysis demonstrates that a single malicious server can compromise the entire agent’s behavior without user awareness.

Maloyan and Namiot analyze three architectural vulnerabilities in the MCP specification itself: absent server identity verification, unauthenticated sampling requests, and lack of isolation between co-installed servers \cite{maloyan2026breaking_protocol}. These are protocol design weaknesses, not implementation bugs. Their evaluation shows MCP amplifies prompt injection success rates by 23--41\% compared to non-MCP baselines. Together, these taxonomies define the attack space that any risk scoring system must cover.

\subsection{Tool Poisoning and Parasitic Attacks}

Tool poisoning---manipulation of tool metadata to steer agent behavior---has emerged as the most effective MCP attack class. Wang et al.\ introduce MCPTox, a benchmark spanning 45 real-world servers and 353 tools \cite{wang2025mcptox}. They evaluate three paradigms: explicit function hijacking, implicit behavioral steering, and cross-server poisoning. Results show 72.8\% overall attack success for explicit poisoning (GPT-4o: 61.8\%, Claude: 34.3\%). Critically, tool poisoning is mechanistically distinct from indirect prompt injection, requiring dedicated detection.

Li et al.\ demonstrate that implicit poisoning is far more dangerous: their MCP-ITP framework achieves 84.2\% attack success while evading detection at a 0.3\% rate \cite{li2026mcp_itp}. Implicitly poisoned tools are never directly invoked---their metadata causes the agent to misuse \emph{other} legitimate tools, making the attack invisible to invocation-level monitoring. For risk scoring, this implies metadata semantics and cross-tool reference patterns must be core inputs.

Zhao et al.\ introduce parasitic toolchain attacks, where no single tool is malicious but dangerous behavior emerges from tool combinations \cite{zhao2025mind_your_server}. Their analysis identifies 472 External Ingestion Tools, 391 Privacy Access Tools, and 180 Network Access Tools, with 27.2\% of servers exposing exploitable combinations and a 90\% real-world attack success rate. Hu et al.\ identify a particularly insidious variant: their Log-To-Leak study shows that benign logging and monitoring tools can be weaponized for covert data exfiltration while the agent’s primary task completes successfully \cite{hu2026log_to_leak}. This means output quality cannot serve as a safety signal---the risk model must evaluate tool combinations and data flow patterns.

Wang et al.\ approach detection from the agent’s internal perspective with MindGuard, inspecting the LLM’s decision-making process to detect when metadata poisoning influences tool selection \cite{wang2026mindguard}. Their identification of the code/data confusion problem---where the LLM treats untrusted metadata as implicit code---suggests that risk scoring may benefit from agent-internal behavioral signals alongside static metadata features.

\subsection{Defense Frameworks and Runtime Guardrails}

Several defense frameworks have been proposed, employing architectures from adaptive trust calibration to multi-agent guardrails. Zhou et al.\ introduce MCPShield, implementing three-stage adaptive trust: pre-invocation probing, execution-time monitoring via sandbox projections, and post-invocation temporal reasoning to detect rug-pull attacks \cite{zhou2026mcpshield}. MCPShield achieves 95.30\% detection across 17 attack types but outputs a binary trusted/untrusted decision.

Xing et al.\ propose MCP-Guard, a defense-in-depth pipeline cascading three stages: fast pattern matching ($<$2ms, catching 38.9\% of threats), a fine-tuned neural classifier ($\sim$55ms), and LLM arbitration for ambiguous cases \cite{xing2026mcp_guard}. The system achieves F1 of 95.4\% with 2.04$\times$ speedup over LLM-only defenses. This cascaded design principle directly informs our multi-stage scoring pipeline, though MCP-Guard’s output remains binary.

Gosmar et al.\ propose a multi-agent NLP framework classifying prompt injection into 10 categories with a four-stage detection pipeline \cite{gosmar2025prompt_injection}. Mou et al.\ introduce ToolSafe, operating at the individual tool-call level with proactive feedback loops enabling agent self-correction rather than outright blocking \cite{mou2026toolsafe}. Xiang et al.\ deploy GuardAgent, a separate guard agent using deterministic code-based enforcement policies, decoupling security from task execution \cite{xiang2025guardagent}.

All five frameworks produce binary or categorical decisions (safe/unsafe, allow/block). None outputs a numeric risk score enabling graduated responses---auto-allow, log-and-monitor, escalate-to-human, or auto-deny---which is the core gap this work addresses.

\subsection{Access Control and Privilege Management for AI Agents}

A complementary line of research addresses constraining what agents are permitted to do. Shi et al.\ introduce Progent, a programmable privilege control system with fine-grained, composable policies at the tool-call level \cite{shi2025progent}. Progent reduces attack success from 41.2\% to 2.2\% but its policies are static and predefined---they cannot adapt to runtime context or the specific risk level of individual requests.

Buhler et al.\ propose AgentBound, treating access control as a first-class MCP primitive through AgentManifest (permission declarations) and AgentBox (OS-level enforcement) \cite{buhler2025agentbound}. Their analysis reveals that MCP servers currently run with full user-level permissions, creating an attack surface far broader than required.

Wu et al.\ address automating data access permissions for AI agents, proposing a Permission Assistant that learns user preferences to generate minimal-privilege policies \cite{wu2025automating_data_access}. All three works enforce binary access boundaries and do not quantify the degree of risk associated with a permitted action. A graduated scoring approach would complement access control by informing policy decisions with risk-proportionate evidence.

\subsection{Risk Scoring, Trust Management, and Vulnerability Prioritization}

The question of how to quantify security risk numerically has been studied in vulnerability management but remains unexplored for MCP. Jafarikhah et al.\ demonstrate that LLMs can predict CVSS base metric scores from vulnerability descriptions alone, achieving approximately 79\% accuracy with two-shot prompting \cite{jafarikhah2026description_to_score}. This establishes that text-based risk scoring is feasible---and directly relevant to MCP, where tool descriptions share structural similarity with CVE descriptions (both describe functionality, access scope, and potential impact in natural language). The key difference is that CVSS scoring is performed once per vulnerability, whereas MCP scoring must operate dynamically at request time.

Raza et al.\ survey Trust, Risk, and Security Management (TRiSM) for agentic AI, analyzing 180 papers and identifying risk quantification as one of six core governance pillars \cite{raza2025trism}. They specify that multi-agent risk signals must be \emph{dynamic} (updated as behavior is observed), \emph{contextual} (sensitive to the operational environment), and \emph{composable} (aggregatable across agents and tools). Despite identifying these requirements, the survey finds no system producing graduated risk scores for MCP or agentic AI tool use. Together, these works provide the methodological foundation but leave the application to MCP tool invocations as an open problem.

\subsection{Positioning of This Work}

Prior work establishes that the MCP ecosystem is large with minimal security oversight \cite{hou2025mcp_landscape,li2025understanding_mcp}, that the attack surface spans tool poisoning (72.8--84.2\% success), protocol exploits (23--41\% amplification), and parasitic compositions (90\% success) with implicit attacks evading detection at 0.3\% \cite{ferrag2025prompt_to_protocol,wang2025mcptox,li2026mcp_itp,zhao2025mind_your_server,hu2026log_to_leak}, that all defenses and access controls produce binary decisions \cite{zhou2026mcpshield,xing2026mcp_guard,shi2025progent,buhler2025agentbound,mou2026toolsafe,xiang2025guardagent}, and that LLM-based risk scoring is feasible but unapplied to MCP \cite{jafarikhah2026description_to_score,raza2025trism}. The critical gap is the absence of a graduated, dynamic risk scoring system.

This work addresses this gap by proposing a framework that produces a 1--10 severity score per tool invocation request, enabling a spectrum of response policies---auto-allow (1--3), log-and-monitor (4--5), escalate-to-human (6--7), and auto-deny (8--10)---that balance security with agent utility based on quantified risk.

\paragraph{Contributions.}
\begin{itemize}
    \item A formal risk taxonomy defining base metrics for MCP tool invocation risk, adapted from CVSS and informed by the attack landscape documented across 20 recent studies.
    \item A multi-stage scoring pipeline combining static tool metadata analysis, contextual enrichment from session behavior and ecosystem baselines, and LLM-based risk judgment to produce graduated 1--10 scores.
    \item An evaluation against existing binary defense baselines demonstrating that graduated scoring enables more nuanced and effective security decisions.
\end{itemize}
