01

ההקשר הוא המפתח

היכולות הבסיסיות של מודל AI גנרטיבי תלויות בפרטי האימון מראש שלו, בנתוני האימון ובארכיטקטורת המודל. כדי לגרום למודלים אלה לביצועים טובים יותר ולשפר את הרלוונטיות והקוהרנטיות שלהם למשימה שלך, עליך לספק להם הקשר טוב.

כאן הכוונה ב"הקשר" היא למידע שהמודל משתמש בו כדי לייצר תגובות רלוונטיות וקוהרנטיות. ההקשר קובע כיצד המודל מבין את השיחה, ממשיך טקסט, או מייצר תמונה.

ניתן לספק הקשר בדרכים שונות, בהתאם לסוג המודל והמשימה:

1. מודלי טקסט (GPT, DeepSeek, LLaMA)

מקבלים הקשר דרך:

  • הקשר פרומפט: טקסט הקלט או השאילתה שמנחה את תגובת המודל.
  • חלון טוקנים: מספר הטוקנים שהמודל יכול "לזכור" בכל רגע (לדוגמה, GPT-4-Turbo יכול לטפל בכ-128K טוקנים).
  • היסטוריית שיחה: בצ'אטבוטים, חילופי הדברים הקודמים עוזרים לשמור על הקשר בדיאלוגים מרובי-תורות.
  • Retrieval-Augmented Generation (RAG): הקשר ממסמכים חיצוניים שנשלפים דינמית כדי לשפר תגובות.

2. מודלי תמונה ומולטי-מודל (DALL·E, Gemini)

מקבלים הקשר דרך:

  • תיאורי טקסט: הפרומפט מנחה את יצירת התמונה.
  • הקשר ויזואלי: אם מסופקת תמונה, המודל מנתח את תוכנה לפני יצירת אלמנטים חדשים.
  • הקשר בין-מודאלי: כאשר משלבים טקסט ותמונות, המודלים מפרשים את שניהם כדי לייצר פלט משמעותי.

3. מודלי יצירת קוד (Codex, DeepSeek-Coder)

מקבלים הקשר דרך:

  • בלוקי קוד קודמים: ההקשר כולל קוד קיים, שמות פונקציות והערות.
  • תחביר שפת תכנות: המודל מבין תבניות ספציפיות לשפה.
  • תיעוד חיצוני: חלק מהמודלים משתמשים ב-APIs ותיעוד להצעות מדויקות יותר.

4. מודלי דיבור ושמע (Whisper, AudioPaLM)

מקבלים הקשר דרך:

  • קטעי שמע: דיבור או מוזיקה קודמים מיידעים את החלק הנוצר הבא.
  • תכונות לשוניות ואקוסטיות: טון, מהירות ואינטונציה משפיעים על תמלול ויצירה.

בקיצור, ההקשר הוא גורם המפתח שמאפשר ל-AI גנרטיבי לייצר פלט רלוונטי וקוהרנטי. ככל שניהול ההקשר טוב יותר, כך ביצועי ה-AI טובים יותר.

עם הזמן, מודלים יכולים לאחזר נתונים אוטומטית כדי לשמש כהקשר. זה נכון במיוחד ל-AI Agents — מערכות שמשתמשות במודלים גנרטיביים בליבה שלהן. כלומר, ה-AI Agents צריכים לחפש מקורות נתונים, לבקש מהמקורות נתונים ספציפיים, וכן הלאה.

כל מקור נתונים (שרת) ממומש בדרכו שלו — לדוגמה, כחבילות קוד פתוח ב-codebase אחר, או כ-JSON RPC להודעות — ולכן אין דרך סטנדרטית עבור מודל AI (לקוח) לחפש ולבקש נתונים. זה מה שנקרא פיצול (fragmentation).

02

בעיית הפיצול לפני MCP

לפני MCP, בניית מערכות AI כרוכה לעיתים קרובות ב:

  • מימושים מותאמים אישית לכל אפליקציית AI להתחבר להקשר הנדרש שלה, מה שהוביל להרבה מאמץ כפול.
  • לוגיקת פרומפטים לא עקבית ושיטות שונות לגישה ולפדרציה של כלים ונתונים בצוותים ובחברות שונות.
  • בעיית N כפול M — מספר גדול של אפליקציות לקוח היה צריך לתקשר עם מספר גדול של שרתים וכלים, מה שיצר רשת מורכבת של אינטגרציות, שכל אחת מהן דרשה עבודת פיתוח ספציפית.

תוצאת הפיצול: שינוי קטן ב-API של GitHub אחד שובר את כל הלקוחות שמחוברים אליו. כל צוות כותב מחדש את אותה לוגיקה. אין אחידות, אין שיתוף פעולה.

03

Model Context Protocol (MCP) — הפתרון

MCP פותר את בעיית הגישה המפוצלת לנתונים. הוא מספק תקן פתוח לחיבור מערכות AI עם מקורות נתונים וכלים — מאגרים, כלים עסקיים, סביבות פיתוח — ומחליף אינטגרציות מפוצלות בפרוטוקול יחיד. בכך, MCP מספק פונגיביליות בין לקוחות ושרתי AI.

MCP פותח על ידי מהש מוראג' (Mahesh Murag) ב-Anthropic ויצא בנובמבר 2024. הוא ממומש כיום ב-Python SDK ו-TypeScript SDK.

MCP מספק דרך סטנדרטית לאפליקציות ל:

  • שיתוף מידע הקשר עם מודלי שפה
  • חשיפת כלים ויכולות למערכות AI
  • בניית אינטגרציות וורקפלואים ניתנים להרכבה

הפרוטוקול משתמש בהודעות JSON-RPC 2.0 לביסוס תקשורת בין:

  • Hosts: אפליקציות LLM שיוזמות חיבורים
  • Clients: מחברים בתוך אפליקציית ה-Host
  • Servers: שירותים שמספקים הקשר ויכולות

כלים פופולריים שתומכים ב-MCP כיום כוללים: Cursor, Windsurf (Codium), Cline (תוסף VS Code), Claude Desktop ו-Claude Code.

MCP שואב השראה מ-Language Server Protocol (LSP), שמתקין כיצד להוסיף תמיכה בשפות תכנות על פני מערכת שלמה של כלי פיתוח. באופן דומה, MCP מתקין כיצד לשלב הקשר וכלים נוספים בתוך מערכת האקולוגית של אפליקציות ה-AI.

04

ארכיטקטורת MCP

MCP עוקב אחר ארכיטקטורת client-host-server שבה כל Host יכול להריץ מספר instances של Client. ארכיטקטורה זו מאפשרת למשתמשים לשלב יכולות AI באפליקציות תוך שמירה על גבולות אבטחה ברורים והפרדת דאגות.

בנוי על JSON-RPC, MCP מספק פרוטוקול session ממדי המתמקד בחילוף הקשר ותיאום דגימה בין לקוחות לשרתים.

Host — המארח

תהליך ה-Host פועל כמיכל ומתאם. הוא:

  • יוצר ומנהל מספר instances של Client
  • שולט בהרשאות חיבור Client ומחזור החיים שלהם
  • אוכף מדיניות אבטחה ודרישות הסכמה
  • מטפל בהחלטות הרשאת המשתמש
  • מתאם אינטגרציית AI/LLM ו-sampling
  • מנהל צבירת הקשר מכל הלקוחות

Clients — הלקוחות

כל Client נוצר על ידי ה-Host ומקיים חיבור שרת מבודד. הלקוח:

  • מקים session ממדי אחד לכל שרת (יחס 1:1)
  • מטפל במשא ומתן על פרוטוקול וחילוף יכולות
  • מנתב הודעות פרוטוקול דו-כיווני
  • מנהל מנויים ותגובות
  • שומר על גבולות אבטחה בין שרתים

אפליקציות MCP Client הן אפליקציות AI או סוכנים שרוצים לגשת למערכות חיצוניות. דוגמאות: אפליקציות first-party של Anthropic, Cursor, Windsurf, וסוכנים כמו Goose. המאפיין המרכזי הוא תאימות MCP — בנוי לתקשר באמצעות ממשקים מתוקנים: prompts, tools ו-resources.

כשמדובר בכלים — מודל השפה בתוך אפליקציית הלקוח מחליט מתי הכי טוב להפעיל כלים שהשרת חשף. למשאבים — לאפליקציית הלקוח יש שליטה על האופן שבו הנתונים שהשרת חשף נמצאים בשימוש. פרומפטים נחשבים ל-user-controlled tools שהמשתמש מפעיל דרך אפליקציית הלקוח.

Servers — השרתים

שרתים מספקים הקשר ויכולות מיוחדות. הם:

  • חושפים resources, tools ו-prompts דרך MCP primitives
  • פועלים באופן עצמאי עם אחריות ממוקדת
  • מבקשים sampling דרך ממשקי לקוח
  • חייבים לכבד אילוצי אבטחה
  • יכולים להיות תהליכים מקומיים או שירותים מרוחקים

שרתי MCP פועלים כעטיפות או מתווכים שמספקים דרך סטנדרטית לגישה למערכות חיצוניות: בסיסי נתונים, CRMs כמו Salesforce, מערכות קבצים מקומיות ומערכות בקרת גרסאות כמו Git.

העיקרון המרכזי: ברגע שנבנה שרת MCP — הוא נגיש לכל לקוח MCP. ברגע שאפליקציה היא MCP-תואמת — היא מתחברת לכל שרת. זה פותר את בעיית N×M.

05

פיצ'רים של שרת MCP

שרתי MCP מספקים את אבני הבנייה הבסיסיות — Prompts, Resources, Tools — להוספת הקשר למודלים דרך MCP. primitives אלה מאפשרים אינטראקציות עשירות בין לקוחות, שרתים ומודלי שפה.

💬 Prompts — תבניות פרומפט

Prompts מאפשרים לשרתים לספק הודעות והוראות מובנות לאינטראקציה עם מודלים. לקוחות יכולים לגלות prompts זמינים, לאחזר את תוכנם ולספק ארגומנטים כדי להתאים אותם אישית.

Prompts מתוכננים להיות user-controlled — נחשפים מהשרתים ללקוחות עם הכוונה שהמשתמש יבחר בהם מרצון לשימוש. בדרך כלל הם מופעלים דרך פקודות שמשתמש מאתחל בממשק — לדוגמה, כפקודות slash.

typescript — הצהרת prompts capability בשרת
const server = new Server({
  name: "my-server",
  version: "1.0.0"
}, {
  capabilities: {
    prompts: { listChanged: true } // חובה להצהיר תמיכה ב-prompts
  }
});

server.setRequestHandler(ListPromptsRequestSchema, async () => ({
  prompts: [{
    name: "code_review",
    description: "בצע סקירת קוד מעמיקה",
    arguments: [
      { name: "language", description: "שפת התכנות", required: true }
    ]
  }]
}));

📦 Resources — משאבי נתונים

Resources מאפשרים לשרתים לשתף נתונים שמספקים הקשר למודלים: קבצים, סכמות בסיסי נתונים, מידע ספציפי לאפליקציות. כל resource מזוהה על ידי URI ייחודי.

Resources ב-MCP מתוכננים להיות application-driven — אפליקציות ה-Host קובעות כיצד לשלב הקשר בהתאם לצרכיהן. לדוגמה, אפליקציות יכולות:

  • לחשוף resources דרך אלמנטים ב-UI לבחירה מפורשת (בתצוגת עץ או רשימה)
  • לאפשר למשתמש לחפש ולסנן resources זמינים
  • ליישם הכללת הקשר אוטומטית, בהתבסס על heuristics או בחירת מודל ה-AI

ה-capability תומך בשני פיצ'רים אופציונליים: subscribe (האם הלקוח יכול להירשם לקבל עדכונים על שינויים ב-resources) ו-listChanged (האם השרת יפלוט התראות כשרשימת ה-resources משתנה).

typescript — חשיפת resources מהשרת
server.setRequestHandler(ListResourcesRequestSchema, async () => ({
  resources: [
    {
      uri: "file:///project/README.md",
      name: "קובץ README",
      mimeType: "text/markdown"
    },
    {
      uri: "db://schema/users",
      name: "סכמת טבלת משתמשים",
      mimeType: "application/json"
    }
  ]
}));

// קריאת תוכן resource
server.setRequestHandler(ReadResourceRequestSchema, async ({ params }) => ({
  contents: [{
    uri: params.uri,
    mimeType: "text/plain",
    text: await readFile(params.uri)
  }]
}));

🔧 Tools — כלים לביצוע

MCP מאפשר לשרתים לחשוף כלים שמודלים יכולים להפעיל. כלים מאפשרים למודלים לתקשר עם מערכות חיצוניות: שאילתות בסיסי נתונים, קריאות API, ביצוע חישובים. כל כלי מזוהה על ידי שם ייחודי וכולל מטא-דאטה המתאר את הסכמה שלו.

כלים ב-MCP מתוכננים להיות model-controlled — מודל השפה יכול לגלות ולהפעיל כלים אוטומטית בהתאם להבנתו ההקשרית ולפרומפטים של המשתמש.

typescript — הגדרת tool ב-TypeScript SDK
server.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: [{
    name: "query_database",
    description: "שלח שאילתת SQL לבסיס הנתונים",
    inputSchema: {
      type: "object",
      properties: {
        sql:  { type: "string", description: "שאילתת SQL להרצה" },
        limit: { type: "number", description: "מגבלת שורות תוצאה" }
      },
      required: ["sql"]
    }
  }]
}));

server.setRequestHandler(CallToolRequestSchema, async ({ params }) => {
  const { name, arguments: args } = params;
  if (name === "query_database") {
    const rows = await db.query(args.sql, args.limit ?? 100);
    return {
      content: [{ type: "text", text: JSON.stringify(rows) }]
    };
  }
});
06

פיצ'רים של לקוח MCP

לקוחות יכולים לממש פיצ'רים נוספים שמעשירים את שרתי MCP המחוברים:

📂 Roots — שורשי מערכת הקבצים

Roots מגדירים את גבולות הפעולה של שרתים בתוך מערכת הקבצים — מאפשרים להם להבין אילו ספריות וקבצים הם יכולים לגשת אליהם. MCP מספק דרך סטנדרטית ללקוחות לחשוף "roots" של מערכת הקבצים לשרתים. שרתים יכולים לבקש את רשימת ה-roots מלקוחות שתומכים בכך, ולקבל התראות כשרשימה זו משתנה.

הגדרת root כוללת:

  • uri: מזהה ייחודי ל-root (חייב להיות file:// URI)
  • name: שם אופציונלי קריא לאנוש למטרות תצוגה
json — דוגמת roots לשימושים שונים
// פרויקט
{ "uri": "file:///home/user/projects/my-app", "name": "My Project" }

// Home directory
{ "uri": "file:///home/user", "name": "Home Directory" }

// ספריית עבודה ספציפית
{ "uri": "file:///var/www/html", "name": "Web Root" }

🔄 Sampling — דגימה מהמודל

MCP מספק דרך סטנדרטית לשרתים לבקש LLM sampling (completions או generations) ממודלים דרך לקוחות. זרימה זו מאפשרת ללקוחות לשמור על שליטה בגישה למודל, בחירה והרשאות, תוך שהיא מאפשרת לשרתים למנף יכולות AI — ללא צורך ב-API keys בצד השרת.

שרתים יכולים לבקש אינטראקציות מבוססות טקסט או תמונה ולכלול אופציונלית הקשר מהשרתים.

Sampling ב-MCP מאפשר לשרתים לממש התנהגויות אג'נטיות, על ידי אפשרות של קריאות LLM מקוננות בתוך פיצ'רים אחרים של שרת MCP.

json — בקשת sampling מהלקוח
{
  "method": "sampling/createMessage",
  "params": {
    "messages": [{
      "role": "user",
      "content": { "type": "text", "text": "נתח את הנתונים הבאים..." }
    }],
    "maxTokens": 1024,
    "systemPrompt": "אתה מנתח נתונים מומחה"
  }
}
07

קומפוזביליות ו-Sampling — הלב של הסוכנים

קומפוזביליות ב-MCP מדגישה שההבחנה בין לקוח לשרת היא לוגית ולא פיזית. זאת אומרת שכל אפליקציה, API, או סוכן יכולים לפעול גם כ-MCP Client וגם כ-MCP Server בו זמנית.

תפקיד כפול זה מאפשר יצירת מערכות שכבתיות ומשורשרות: משתמש מתקשר עם אפליקציית סוכן ראשי (לקוח), שאז מתקשר עם תת-סוכן מיוחד (כשרת). תת-סוכן זה, בתורו, יכול לפעול כלקוח ולהפעיל שרתי MCP אחרים (כמו שרת מערכת קבצים או שרת חיפוש אינטרנט) כדי למלא את משימתו.

רלוונטיות לסוכנים: קומפוזביליות היא קריטית לבניית ארכיטקטורות סוכן מודולריות ומתקדמות. היא מאפשרת יצירת מערכות היררכיות של סוכנים, שבהן סוכנים שונים יכולים להתמחות במשימות ספציפיות ולהאציל תת-משימות לסוכנים אחרים.

בשילוב, sampling וקומפוזביליות הם מאפשרים חזקים לסוכנים מתקדמים:

  • חלוקת אינטליגנציה על פני מערכת רב-סוכנים, עם הלקוח ששולט באינטראקציות ה-LLM בפועל
  • בניית מערכות סוכן מרובות-שכבות מורכבות שבהן סוכנים מיוחדים עובדים יחד
  • גמישות ומודולריות מוגברות בעיצוב סוכנים — יכולות חדשות ניתן לשלב בקלות
  • פוטנציאל לסוכנים שמתפתחים ומסתגלים על ידי אינטראקציה עם סוכנים ושירותים אחרים
08

מחזור חיי חיבור

MCP מגדיר מחזור חיים קפדני לחיבורי client-server המבטיח משא ומתן נכון על יכולות וניהול מצב.

שלושה שלבים בחיבור: Initialization (אתחול) ← Operation (פעולה) ← Shutdown (כיבוי)

שלב 1: Initialization — אתחול

שלב האתחול חייב להיות האינטראקציה הראשונה בין לקוח לשרת. הלקוח מאתחל על ידי שליחת בקשת initialize המכילה:

  • גרסת פרוטוקול נתמכת
  • יכולות הלקוח
  • מידע על מימוש הלקוח

השרת חייב להגיב עם יכולותיו ומידעו. לאחר אתחול מוצלח, הלקוח שולח התראת initialized.

json — initialize request (לקוח ← שרת)
// שלב 1: הלקוח שולח
{
  "jsonrpc": "2.0", "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2024-11-05",
    "capabilities": {
      "sampling": {},
      "roots": { "listChanged": true }
    },
    "clientInfo": { "name": "ClaudeDesktop", "version": "1.0.0" }
  }
}

// שלב 2: השרת מגיב
{
  "jsonrpc": "2.0", "id": 1,
  "result": {
    "protocolVersion": "2024-11-05",
    "capabilities": {
      "tools": { "listChanged": true },
      "resources": { "subscribe": true }
    },
    "serverInfo": { "name": "my-mcp-server", "version": "0.9.2" }
  }
}

// שלב 3: הלקוח מודיע שמוכן
{ "jsonrpc": "2.0", "method": "notifications/initialized" }

שלב 2: משא ומתן על גרסה

בבקשת initialize, הלקוח שולח גרסת פרוטוקול שהוא תומך בה (בדרך כלל העדכנית). אם השרת תומך בגרסה המבוקשת — הוא מגיב עם אותה גרסה. אחרת, השרת מגיב עם גרסה אחרת שהוא תומך בה. אם הלקוח לא תומך בגרסת תגובת השרת — הוא צריך להתנתק.

שלב 3: Operation — פעולה שוטפת

שני הצדדים מכבדים את גרסת הפרוטוקול שנקבעה ומשתמשים רק ביכולות שהוסכמו בהצלחה.

שלב 4: Shutdown — כיבוי מסודר

עבור stdio transport, הלקוח מאתחל כיבוי על ידי: סגירת input stream לתהליך הילד (השרת) ← המתנה לשרת לצאת ← שליחת SIGTERM אם השרת לא יצא בזמן סביר ← שליחת SIGKILL אם עדיין לא יצא.

עבור HTTP transports — הכיבוי מסומן על ידי סגירת חיבורי ה-HTTP המשויכים.

09

אבטחה, אמון ובטיחות

MCP מאפשר יכולות חזקות דרך נתיבי גישה לנתונים שרירותיים וביצוע קוד. עם כוח זה מגיעים שיקולי אבטחה ואמון חשובים שכל המממשים חייבים לטפל בהם בקפידה.

עקרונות מפתח של אבטחת MCP

✅ הסכמה ושליטת משתמש

המשתמשים חייבים להסכים במפורש לכל גישה לנתונים וכל פעולה, ולהבין אותן. הם חייבים לשמור על שליטה על אילו נתונים משותפים ואילו פעולות מתבצעות.

🔒 פרטיות נתונים

Hosts חייבים לקבל הסכמת משתמש מפורשת לפני חשיפת נתוני משתמש לשרתים. נתוני משתמש צריכים להיות מוגנים עם בקרות גישה מתאימות.

⚠️ בטיחות Tools

כלים מייצגים ביצוע קוד שרירותי וחייבים להיות מטופלים בזהירות מתאימה. Hosts חייבים לקבל הסכמת משתמש מפורשת לפני הפעלת כל כלי.

🎛️ שליטה ב-LLM Sampling

המשתמשים חייבים לאשר במפורש כל בקשת sampling. הפרוטוקול מגביל בכוונה את הנראות של השרת לפרומפטים.

הנחיות מימוש

מממשי MCP צריכים לבנות זרימות הסכמה והרשאה חזקות, לספק תיעוד ברור של השלכות האבטחה, ליישם בקרות גישה ומגני נתונים מתאימים, לעקוב אחר שיטות עבודה מומלצות לאבטחה ולשקול השלכות פרטיות בעיצובי פיצ'רים.

שים לב: אימות והרשאה אינם כרגע חלק מהמפרט הליבה של MCP, אבל צפויים להיות מוכנסים בקרוב. OAuth 2.0 כבר בדרך.

MCP Registry API — כרגע בפיתוח

מטרת MCP Registry API שנמצא כרגע בפיתוח היא לספק דרך מרוכזת לגלות, לאמת ולנהל שרתי MCP.

האתגרים המרכזיים שה-Registry API שואף לפתור:

  • יכולת גילוי (Discoverability): כיום, מציאת שרתי MCP רלוונטיים מפוצלת. ה-Registry ישמש כשירות מטא-נתונים מאוחד שמקל על משתמשים ואפליקציות לאתר שרתים זמינים ויכולותיהם.
  • מידע פרוטוקול: ה-Registry יספק מידע על הפרוטוקול שבו שרת משתמש (לדוגמה, standard IO או SSE) ומיקומו (קובץ מקומי או URL מרוחק).
  • אמון ואימות: אתגר משמעותי הוא קביעת אמינות ואותנטיות של שרתי MCP. ה-Registry שואף לטפל בכך על ידי הצעת מנגנוני אימות — לדוגמה, ציון שרתים רשמיים מחברות מכובדות כמו Shopify או Grafana.
  • פרסום: ה-Registry יפשט את תהליך פרסום שרתי MCP חדשים לקהל רחב יותר.
  • ניהול גרסאות: ה-Registry יספק דרך לנהל ולעקוב אחר גרסאות שונות של שרתים.
  • ניהול מטא-נתונים: ה-Registry יאחסן מטא-נתונים על שרתי MCP כולל יכולותיהם (resources ו-tools).
12

שרתים מרוחקים ואינטגרציית OAuth 2.0

חשיבות השרתים המרוחקים

כיום, MCP מסתמך לעיתים קרובות על תקשורת מקומית באמצעות standard IO, שעלולה להכניס חיכוך מבחינת הגדרה ופריסה. שרתים מרוחקים, באמצעות פרוטוקולים כמו SSE (Server-Sent Events), מאפשרים לשרתי MCP להתארח בכתובות URL ציבוריות — ולהיות נגישים דרך האינטרנט.

פיתוח זה מסיר את הצורך ממשתמשים להבין את המורכבויות של אירוח שרתי MCP. כמו גישה לאתר אינטרנט, משתמשים יוכלו להתחבר לשרת MCP מרוחק פשוט דרך URL.

אינטגרציית OAuth 2.0

אינטגרציית OAuth 2.0 מספקת מנגנון מתוקן ומאובטח לאימות והרשאה בין לקוחות MCP לשרתים מרוחקים.

  • MCP כעת תומך ב-OAuth 2.0 handshake שבו השרת מתזמר את זרימת האימות, מתקשר עם ספק ה-OAuth (לדוגמה, Slack).
  • הלקוח (המשתמש) מרשה את החיבור דרך זרימה מוכרת מבוססת-web.
  • לאחר אימות, השרת מחזיק את ה-OAuth token ויכול לספק ללקוח session token לאינטראקציות עתידיות.
  • מנגנון אימות מאובטח זה קריטי לאפשר גישה לנתונים ופונקציונליות רגישים.

השפעה על נגישות ושמישות

ההתקדמויות הללו צפויות להשפיע לטובה בצורה עמוקה על הנגישות והשמישות של שרתי MCP:

  • הפחתת מחסום כניסה: קלות החיבור לשרתים מרוחקים דרך URLs מורידה משמעותית את המחסום הטכני — גם עבור מפתחים שרוצים לנצל יכולות MCP וגם עבור משתמשי קצה.
  • מגוון רחב יותר של אפליקציות: היכולת לאחסן שרתים מרחוק פותחת אפשרויות למגוון רחב יותר של אפליקציות — web apps, mobile apps ו-cloud services.
  • זמינות מוגברת של שרתים: החיכוך המופחת בפיתוח ופריסה צפוי להוביל להתרבות שרתי MCP עם פונקציונליות מגוונת.
  • אבטחה ואמון משופרים: אימוץ OAuth 2.0 מספק תקן מוכר לגישה מאובטחת למשאבים מרוחקים — חיוני לבניית אמון משתמשים.
13

סוכנים שמתפתחים בעצמם

רישום שרתי MCP מאפשר לסוכנים לגלות יכולות ונתונים חדשים בזמן אמת — מבלי שתוכנתו עם אותו ידע מראש. כלומר, סוכן יכול להיתקל עם משימה חדשה, לדרוש גישה למקור נתונים שלא ידע עליו, ולחפש באופן יזום כלים הנחוצים לו.

שקלו את הדוגמה של סוכן קידוד כללי שמוטל עליו לבדוק לוגים של Grafana: אם הסוכן לא הוגדר מראש עם ספציפיות של שרת Grafana, הוא יכול לתקשר עם ה-Registry, לחפש שרת Grafana רשמי ומאומת עם ה-APIs הנחוצים, להתחבר אליו (דרך SSE) ולהמשיך לשאול לוגים ולטפל בבאג.

יתרונות פוטנציאליים של סוכנים שמתפתחים בעצמם

  • הסתגלות משופרת: יתרון משמעותי הוא יכולת הסוכן המוגברת להסתגל למצבים ומשימות חדשות. במקום להיות מוגבל ליכולות המתוכנתות מראש, הוא יכול ללמוד ולהרחיב את הפונקציונליות שלו לפי הצורך.
  • מגוון רחב יותר של יישומים: סוכנים שמתפתחים יכולים להתמודד עם מגוון משימות רחב יותר ללא צורך מפתחים לצפות כל תרחיש אפשרי.
  • חוויית משתמש משופרת: משתמשים יכולים לתקשר עם סוכנים כלליים יותר שיכולים לטפל בבקשות מגוונות.
  • שיפור מתמיד: סוכנים יכולים "ללמוד" ולהשתפר עם הזמן על ידי גילוי וכלול כלים חדשים ויעילים יותר.
  • עומס פיתוח מופחת: מפתחים יכולים להתמקד בבניית לוגיקת חשיבה ביצוע המשימות, בסמיכות ל-Registry לספק גישה לאקוסיסטם הולך וגדל.

שיקולים סביב סוכנים שמתפתחים בעצמם

  • ממשל ואבטחה: שיקול קריטי הוא שליטה על אילו שרתים ויכולות סוכן יכול לגשת. ללא אמצעי הגנה נכונים, סוכן שמתפתח עלול להתחבר לשרתים לא מהימנים או זדוניים.
  • אמון ואימות: הבטחת אמינות השרתים שהתגלו היא עיקרית. ה-Registry שואף לטפל בכך דרך מנגנוני אימות.
  • ביצועים ויעילות: תהליך גילוי ושילוב שרתים חדשים באופן דינמי עלול להכניס עיכובים. אופטימיזציה תהיה חשובה.
  • בחירת כלי וחשיבה: ייתכן שיהיה קשה לוודא שסוכן מקבל החלטות חכמות לגבי אילו כלים לבחור מתוך המספר הגדול האפשרי.
  • ניפוי שגיאות ומעקב: ככל שסוכנים מורכבים יותר ומסתמכים על רכיבים שהתגלו דינמית, ניפוי שגיאות ומעקב על התנהגותם עלולים להיות מאתגרים יותר.
  • ניהול גרסאות ותאימות: ככל שמתפתחים שרתים ממשקי ה-API שלהם משתנים, יש צורך במנגנוני גרסאות ב-Registry.
14

עקרונות העיצוב של MCP

MCP בנוי על מספר עקרונות עיצוב מרכזיים שמיידעים את הארכיטקטורה ויישומו:

עיקרון 1: שרתים צריכים להיות פשוטים מאוד לבנייה. אפליקציות Host מטפלות באחריויות התיאום המורכבות. שרתים מתמקדים ביכולות ספציפיות ומוגדרות היטב. ממשקים פשוטים ממזערים תקורות מימוש. הפרדה ברורה מאפשרת קוד ניתן לתחזוקה.

עיקרון 2: שרתים צריכים להיות ניתנים להרכבה מאוד (highly composable). כל שרת מספק פונקציונליות ממוקדת בבידוד. ניתן לשלב מספר שרתים בצורה חלקה. פרוטוקול משותף מאפשר יכולת הפעלה הדדית. עיצוב מודולרי תומך בהרחבה.

עיקרון 3: שרתים לא צריכים לקרוא את השיחה השלמה, ולא "להציץ" לשרתים אחרים. שרתים מקבלים רק מידע הקשר הנחוץ. היסטוריית שיחה מלאה נשארת עם ה-Host. כל חיבור שרת שומר על בידוד. תהליך ה-Host אוכף גבולות אבטחה.

עיקרון 4: ניתן להוסיף פיצ'רים לשרתים וללקוחות בהדרגה. הפרוטוקול הליבה מספק פונקציונליות מינימלית הנדרשת. יכולות נוספות ניתנות למשא ומתן לפי הצורך. שרתים ולקוחות מתפתחים באופן עצמאי. הפרוטוקול מתוכנן לנרחבות עתידית. נשמרת תאימות לאחור.

סיכום — עקרונות הפרוטוקול

MCP מגדיר שלושה סוגי הודעות ליבה בהתבסס על JSON-RPC 2.0:

  • Requests: הודעות דו-כיווניות עם method וparameters המצפות לתגובה
  • Responses: תוצאות מוצלחות או שגיאות שתואמות ל-request IDs ספציפיים
  • Notifications: הודעות חד-כיווניות שאינן דורשות תגובה
json — שלושת סוגי ההודעות
// Request — מצפה לתגובה
{ "jsonrpc": "2.0", "id": 42, "method": "tools/call", "params": { ... } }

// Response — תגובה מוצלחת
{ "jsonrpc": "2.0", "id": 42, "result": { "content": [...] } }

// Response — שגיאה
{ "jsonrpc": "2.0", "id": 42, "error": { "code": -32600, "message": "Invalid Request" } }

// Notification — חד-כיווני, ללא תגובה
{ "jsonrpc": "2.0", "method": "notifications/tools/list_changed" }

מקורות מידע נוספים: התיעוד הרשמי המלא זמין בכתובת spec.modelcontextprotocol.io. מפרט ה-TypeScript Schema הוא מקור האמת לכל הודעות ומבני הפרוטוקול.
