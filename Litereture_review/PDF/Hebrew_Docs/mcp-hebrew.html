Model Context Protocol
המדריך המלא

כיצד MCP פותר את בעיית הפיצול בבניית מערכות AI, ומאפשר לכל סוכן לתקשר עם כל מקור מידע בשפה אחת אחידה

01ההקשר הוא המפתח

היכולות הבסיסיות של מודל AI תלויות בנתוני האימון ובארכיטקטורת המודל. אבל כדי שהמודל יתן תשובות רלוונטיות וקוהרנטיות למשימה שלך, חייבים לספק לו הקשר טוב.

הקשר (Context) הוא המידע שהמודל משתמש בו כדי לייצר תגובה. בלעדיו — המודל מנחש.

📝

מודלי טקסט (GPT, LLaMA)

מקבלים הקשר דרך פרומפט, היסטוריית שיחה, RAG וחלון טוקנים (עד 128K בGPT-4-Turbo)

🖼️

מודלים מולטי-מודליים

טקסט + תמונה כהקשר. המודל מנתח תמונות שנשלחות, ומשלב עם הפרומפט

💻

מודלי קוד (Codex, DeepSeek)

קוד קיים, הערות, שמות פונקציות ותיעוד חיצוני — כולם הקשר

🎙️

מודלי שמע (Whisper)

קטעי שמע קודמים, טון ומהירות דיבור — משפיעים על הדקלום והיצירה

02הבעיה: פיצול ו-N×M

AI Agents מתוכנתים לחפש מידע ממקורות חיצוניים: בסיסי נתונים, APIs, קבצים, כלים — כולם שרתים שמחכים להישאל. אבל לכל שרת יש ממשק שונה.

בעיית N×M

אם יש N אפליקציות AI ו-M שרתי נתונים — צריך לכתוב N×M אינטגרציות שונות. שינוי אחד בשרת אחד שובר את כל הלקוחות שמחוברים אליו.

💡

לפני MCP: כל אפליקציה AI כתבה קוד ייחודי לכל מקור מידע. שינוי API אחד יכל לשבור עשרות אינטגרציות. MCP פותר זאת עם פרוטוקול אחיד.

03MCP — הפתרון

Model Context Protocol (MCP) הוא תקן פתוח שפותח על ידי Anthropic ויצא לאור בנובמבר 2024. הוא מספק דרך אחידה לחבר מערכות AI למקורות מידע וכלים, ומחליף את הפיצול בפרוטוקול יחיד.

04ארכיטקטורת MCP

MCP מבוסס על ארכיטקטורת Host–Client–Server. כל Host יכול להריץ מספר Client-ים, וכל Client מחובר בחיבור 1:1 לשרת.

🏠 Host

מנהל את ה-Clients, מאשר חיבורים, מיישם מדיניות אבטחה, ומתכלל הקשר מכלל השרתים.

🔌 Client

יוצר חיבור 1:1 עם שרת אחד. מנהל משא ומתן על יכולות (Capabilities) ומנתב הודעות.

🖥️ Server

חושף כלים, משאבים ופרומפטים. פועל בהיגיון ממוקד ועצמאי — יכול להיות תהליך מקומי או שירות מרוחק.

05שלושת ה-Primitives

כל שרת MCP חושף שלושה סוגי יכולות עיקריות:

💬 Prompts — תבניות הנחיה

תבניות מוכנות מראש שמנחות את המודל. המשתמש בוחר בהן מה-UI (לדוגמה: כפקודת slash). הן user-controlled — המשתמש מפעיל אותן במודע.

📄 דוגמה — הצהרת Prompt בשרת
const server = new MCPServer();

server.setRequestHandler("prompts/list", async () => ({
  prompts: [
    {
      name: "code_review",
      description: "סקירת קוד מקצועית",
      arguments: [
        { name: "language", required: true },
        { name: "focus", required: false }
      ]
    }
  ]
}));

📦 Resources — משאבי מידע

נתונים מובנים שמספקים הקשר: קבצים, סכמות DB, תיעוד. כל משאב מזוהה על ידי URI ייחודי. הם application-controlled — האפליקציה מחליטה מתי לטעון אותם.

📄 דוגמה — חשיפת Resource
server.setRequestHandler("resources/list", async () => ({
  resources: [
    {
      uri: "file:///project/README.md",
      name: "Project README",
      mimeType: "text/markdown"
    },
    {
      uri: "db://schema/users",
      name: "טבלת משתמשים",
      mimeType: "application/json"
    }
  ]
}));

🔧 Tools — כלים לביצוע

פונקציות שהמודל יכול להפעיל: שאילתות DB, קריאות API, חישובים. הם model-controlled — המודל מחליט מתי לקרוא להם בהתאם להקשר.

📄 דוגמה — הגדרת Tool
server.setRequestHandler("tools/list", async () => ({
  tools: [
    {
      name: "query_database",
      description: "שאל את בסיס הנתונים",
      inputSchema: {
        type: "object",
        properties: {
          sql: { type: "string", description: "SQL query" }
        },
        required: ["sql"]
      }
    }
  ]
}));

// טיפול בקריאה לכלי
server.setRequestHandler("tools/call", async ({ params }) => {
  const { name, arguments: args } = params;
  if (name === "query_database") {
    const result = await db.query(args.sql);
    return { content: [{ type: "text", text: JSON.stringify(result) }] };
  }
});

06מחזור חיי חיבור

כל חיבור Client–Server עובר שלושה שלבים:

1. Initialization — אתחול

הClient שולח initialize עם גרסת הפרוטוקול ויכולותיו. השרת מגיב עם יכולותיו. לאחר מכן הClient שולח initialized להכריז שמוכן.

2. Operation — פעולה רגילה

החלפת הודעות לפי היכולות שסוכמו. שני הצדדים מכבדים את גרסת הפרוטוקול ומשתמשים רק ביכולות שהוסכמו.

3. Shutdown — סגירה מסודרת

הClient סוגר את ה-input stream לתהליך הילד, ממתין ליציאתו. אם לא יוצא — שולח SIGTERM, ולאחר מכן SIGKILL.

📄 דוגמה — initialize request
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2024-11-05",
    "capabilities": {
      "sampling": {},       // הClient תומך בsampling
      "roots": { "listChanged": true }
    },
    "clientInfo": {
      "name": "ClaudeDesktop",
      "version": "1.0.0"
    }
  }
}

07שרת MCP מלא — דוגמה מעשית

הנה שרת MCP פשוט ב-Python SDK שחושף כלי חיפוש בקבצים:

🐍 python — simple_mcp_server.py
from mcp.server import Server
from mcp.server.stdio import stdio_server
from mcp.types import Tool, TextContent
import os, json

app = Server("file-search-server")

# חשיפת הכלים הזמינים
@app.list_tools()
async def list_tools():
    return [
        Tool(
            name="search_files",
            description="חפש קבצים לפי שם בתיקייה",
            inputSchema={
                "type": "object",
                "properties": {
                    "path": {"type": "string", "description": "נתיב לחיפוש"},
                    "pattern": {"type": "string", "description": "תבנית חיפוש"}
                },
                "required": ["path", "pattern"]
            }
        )
    ]

# טיפול בהפעלת כלי
@app.call_tool()
async def call_tool(name: str, arguments: dict):
    if name == "search_files":
        path = arguments["path"]
        pattern = arguments["pattern"]
        results = []
        for root, dirs, files in os.walk(path):
            for f in files:
                if pattern in f:
                    results.append(os.path.join(root, f))
        return [TextContent(type="text", text=json.dumps(results, ensure_ascii=False))]

# הפעלת השרת על stdio
async def main():
    async with stdio_server() as (read, write):
        await app.run(read, write, app.create_initialization_options())

if __name__ == "__main__":
    import asyncio
    asyncio.run(main())

08יתרונות לפי קהל יעד

🧑‍💻 מפתחי אפליקציות אפס עבודה נוספת לחיבור שרת חדש אפליקציה MCP-תואמת מתחברת לכל שרת ללא קוד נוסף
🔌 ספקי API/כלים בנה פעם אחת, ראה אימוץ בכל מקום שרת MCP אחד נגיש לכל הלקוחות התואמים
👤 משתמשי קצה אפליקציות AI עשירות יותר גישה חלקה לנתונים ולכלים שהמשתמש כבר מכיר
🏢 ארגונים פיתוח AI סטנדרטי ומהיר הפרדת אחריות בין צוותים, פחות כפילות

09אבטחה ואמון

MCP מאפשר גישה לנתונים וביצוע קוד. זה כוח גדול — ועמו אחריות גדולה:

✅

הסכמת המשתמש

כל גישה לנתונים וכל פעולה חייבת אישור מפורש מהמשתמש

🔒

פרטיות נתונים

Host לא יכול להעביר נתוני משתמש לשרתים ללא הסכמה

⚠️

בטיחות Tools

כלים = ביצוע קוד שרירותי. יש לאשר כל כלי בנפרד

🎛️

שליטה ב-Sampling

המשתמש שולט על מה שנשלח למודל ועל מה השרת רואה

10העתיד: Registry ו-Remote Servers

כיום MCP משתמש בתקשורת מקומית (stdio). אבל Remote Servers דרך SSE יאפשרו חיבור לשרתים דרך URL בלבד — בלי התקנה מקומית.

🗂️ MCP Registry API (בפיתוח)

רישום מרכזי שיאפשר לסוכן AI לגלות שרתים חדשים בזמן אמת. תארו לעצמכם: הסוכן מקבל משימה → מחפש ב-Registry → מוצא שרת Grafana רשמי ומאומת → מתחבר → מבצע.

🚀

אינטגרציית OAuth 2.0 תאפשר אימות מאובטח לשרתים מרוחקים. השרת ינהל את זרימת ה-OAuth, והלקוח יקבל session token לאחר האישור.

✦סיכום

MCP הוא שפת האם של סוכני AI. הוא פותר בעיה ישנה — כל לקוח כתב אינטגרציה ייחודית לכל שרת — עם פרוטוקול פתוח ואחיד מבוסס JSON-RPC 2.0.

עם שלושת ה-Primitives (Prompts, Resources, Tools), ארכיטקטורת Host–Client–Server, ותמיכה ב-Python ו-TypeScript SDK — MCP הפך לתשתית שעליה נבנים סוכני ה-AI החכמים של העתיד.
