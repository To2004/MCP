# Data Organization

This document defines how research data and artifacts are organized in
`Literature_review/` and related locations. The guiding principle:
**separate raw from processed, and document relationships**.

## Core Principles

1. **Raw data is immutable.** Never edit a downloaded PDF. Work on copies.
2. **Generated artifacts live in their own folders.** Summaries, figures,
   and extracted data go in separate folders from sources.
3. **Every folder has a README.** Describe what the folder holds and how
   to navigate it.
4. **Name files so they sort meaningfully.** Numeric prefixes enforce
   reading order.

## `Literature_review/` Structure

```
Literature_review/
├── PDF/                       # Raw source papers (immutable)
│   ├── 1_MCP_Security/
│   │   ├── Score_10/          # Grouped by relevance score
│   │   ├── Score_09/
│   │   └── README.md
│   ├── 2_MCP_Protocol/
│   ├── 3_Multi_Agent_Trust/

│   ├── 5_LLM_Guardrails/

│   ├── 7_Not_Academic/

```

## Raw vs Generated Separation

| Type | Location | Mutable? | Git status |
|------|----------|----------|------------|
| Source PDFs | `Literature_review/PDF/` | No — never edit | Committed |
| Paper summaries | `Literature_review/reviews/paper_summaries/` | Yes — updated as notes grow | Committed |
| Figures | `Literature_review/reviews/figures/` | Yes — regenerated from scripts | Committed |
| Tracking sheets | `Literature_review/Excels/` | Yes | Committed |
| Processing scripts | `Literature_review/scripts/` | Yes | Committed |

**Rule:** if a file can be regenerated from raw inputs + scripts, document
how in the folder's README.

## Topic Numbering

Folders under `PDF/` use `N_TopicName` pattern:

| Number | Topic |
|--------|-------|
| 1 | MCP Security |
| 2 | MCP Protocol |
| 3 | Multi-Agent Trust |
| 4 | Prompt Injection |
| 5 | LLM Guardrails |
| 6 | Risk Scoring |
| 7 | Not Academic / Unmatched |

The number reflects **thematic priority** for the thesis, not chronology.

## Score Buckets

Inside each topic folder, papers are grouped by relevance score:

- `Score_10/` — highest relevance
- `Score_09/`, `Score_08/`, ... down to `Score_00/`
- Score is assigned during literature review (see
  `Literature_review/Excels/`)

Each score folder has its own README listing papers in that bucket.

## Summary File Naming

Paper summaries use the pattern: `NN_Paper_Name_summary_LANG.md`

Examples:
- `01_MCP_Landscape_summary_HE.md` (Hebrew summary of paper 01)
- `02_MCPShield_summary_HE.md`

The numeric prefix matches the paper's ID in the tracking spreadsheet.
`_HE` = Hebrew, `_EN` = English (if both exist).

## Figures and Generated Images

All figures go in `Literature_review/reviews/figures/` with:

- Descriptive filename: `risk_score_distribution_2026.png`
- A README entry listing each figure and how it was generated
- Source script referenced (e.g., "generated by `scripts/plot_scores.py`")

## Never Commit

These should be in `.gitignore`:

- `*.tmp`, `*.bak` — temporary work
- Personal notes in scratch folders
- Datasets >10MB (use external storage or Git LFS)
- Output files from experimental scripts (unless they are the deliverable)

## README-in-Folder Template

Every data folder's README should contain:

```markdown
# <Folder Name>

## Purpose
<One sentence: what this folder exists for>

## Contents
<List of files/subfolders with one-line descriptions>

## How Files Relate
<Explain dependencies: which files are raw, which are generated from what>

## How to Navigate
<Where to start, what order to read>
```

## When to Add a New Category

Create a new topic folder or score bucket only when:

1. You have ≥3 items that belong together
2. The category is distinct from existing ones
3. You update the parent folder's README
