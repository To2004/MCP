## 1. הבעיה המרכזית — למה ניקוד פגיעויות ידני לא מספיק?

### תובנת המפתח

מערכת הניקוד CVSS (Common Vulnerability Scoring System) היא הסטנדרט העולמי להערכת חומרת פגיעויות תוכנה. הבעיה: **תהליך הניקוד הוא ידני, סובייקטיבי, ואיטי** — ובזמן שאנליסטים מנקדים בקצב איטי, פגיעויות חדשות נערמות בקצב מהיר.

### מטאפורה: "מיון בחדר מיון"

דמיינו בית חולים (מאגר NVD) שמקבל חולים (פגיעויות CVE) ללא הפסקה:

```
🏥 בית החולים (NVD — National Vulnerability Database)
    │
    ├── 2023: נכנסו 28,818 חולים (CVEs)
    │   └── צוות של 21 רופאים (אנליסטים) מנסה למיין את כולם
    │
    ├── 2024: נכנסו 40,009 חולים (+38%!)
    │   └── באמצע 2024: פחות מ-10% מ-12,700 חולים קיבלו טיפול מאז פברואר
    │
    └── ⚠️ התור גדל → חולים קריטיים ממתינים → סיכון עצום!
```

**הפתרון שנבדק:** להשתמש ב-LLMs (מודלי שפה גדולים) כ-**"מכונת מיון אוטומטית"** —
המודל קורא את **תיאור הפגיעות** (טקסט חופשי) ומנסה לנחש את **ציוני ה-CVSS** (8 מטריקות בסיס).

### למה הבעיה דחופה — המספרים

| שנה | כמות CVEs חדשים | צוות NVD | יחס (CVEs לאנליסט) |
|-----|-----------------|----------|---------------------|
| 2023 | 28,818 | 21 | ~1,372 לאנליסט |
| 2024 | 40,009 | 21 | ~1,905 לאנליסט |

גורמים מחמירים נוספים:
- **גל הגשות** — כמות ההגשות עקפה את הקצב הידני
- **מעבר סכמה** — המעבר לפורמט JSON חדש עיכב צינורות עבודה
- **איכות הגשות** — רשומות כפולות ודלות באיכות בולעות זמן אנליסט
- **כוח אדם** — אין גידול בצוות למרות הגידול בנפח

## 2. רקע — מהי מערכת CVSS ואיך היא עובדת?

### 2.1 שלוש קבוצות מטריקות

מערכת CVSS v3.1 (של FIRST.Org) מורכבת משלוש קבוצות:

```
┌─────────────────────────────────────────────────────────────────┐
│                      CVSS v3.1 Framework                        │
│                                                                 │
│  ┌──────────────────┐  ┌──────────────────┐  ┌───────────────┐ │
│  │   Base Metrics    │  │ Temporal Metrics  │  │ Environmental │ │
│  │   (מטריקות בסיס)  │  │  (מטריקות זמן)   │  │   (סביבתי)    │ │
│  │                   │  │                  │  │               │ │
│  │ Exploitability:  │  │ • Exploit Code   │  │ • CIA          │ │
│  │ • Attack Vector  │  │   Maturity       │  │   Requirements│ │
│  │ • Attack Complex.│  │ • Remediation    │  │ • Modified     │ │
│  │ • Privileges Req.│  │   Level          │  │   Base Metrics│ │
│  │ • User Interact. │  │ • Report         │  │               │ │
│  │                   │  │   Confidence     │  │               │ │
│  │ Impact:          │  │                  │  │               │ │
│  │ • Scope          │  │ מכפילים:         │  │ מכפילים:      │ │
│  │ • Confidentiality│  │ 0.91 עד 1.00     │  │ High=1.5      │ │

│  │ • Availability   │  │                  │  │               │ │
│  └──────────────────┘  └──────────────────┘  └───────────────┘ │
│                                                                 │
│  ציון סופי: 0 (נמוך) ─────────────────────── 10 (קריטי)        │
└─────────────────────────────────────────────────────────────────┘
```

**המחקר מתמקד ב-8 מטריקות הבסיס בלבד** — אלו הן המטריקות שנגזרות מהתכונות הפנימיות של הפגיעות (לא תלויות הקשר ארגוני).

### 2.2 שמונה מטריקות הבסיס — טבלת פירוט

| מטריקה | קיצור | ערכים אפשריים (עם ציון מספרי) | סוג |
|--------|--------|-------------------------------|-----|
| **Attack Vector** (וקטור התקפה) | AV | N (Network, 0.85), A (Adjacent, 0.62), L (Local, 0.55), P (Physical, 0.20) | Exploitability |
| **Attack Complexity** (מורכבות התקפה) | AC | L (Low, 0.77), H (High, 0.44) | Exploitability |
| **Privileges Required** (הרשאות נדרשות) | PR | N (None, 0.85), L (Low, 0.62/0.68), H (High, 0.27/0.50) | Exploitability |
| **User Interaction** (אינטראקציית משתמש) | UI | N (None, 0.85), R (Required, 0.62) | Exploitability |
| **Scope** (טווח השפעה) | S | U (Unchanged), C (Changed) | Impact Modifier |
| **Confidentiality** (סודיות) | C | H (High, 0.56), L (Low, 0.22), N (None, 0.00) | Impact |
| **Integrity** (שלמות) | I | H (High, 0.56), L (Low, 0.22), N (None, 0.00) | Impact |
| **Availability** (זמינות) | A | H (High, 0.56), L (Low, 0.22), N (None, 0.00) | Impact |

### 2.3 נוסחת חישוב ה-Exploitability

ציון ה-Exploitability (כמה קל לנצל) מחושב כך:

```
Exploitability = 8.22 × AV × AC × PR × UI
```

כלומר, ציון גבוה = קל לניצול (רשת, מורכבות נמוכה, בלי הרשאות, בלי אינטראקציה).

## 3. מתודולוגיה — עיצוב המחקר

### 3.1 מטרת המחקר

לבחון באופן שיטתי האם LLMs כלליים (General-Purpose) יכולים **לקרוא תיאור טקסטואלי של פגיעות ולנבא נכונה את 8 מטריקות ה-CVSS** — בלי שום מידע נוסף.

### 3.2 מטאפורה: "הבוחן העיוור"

```
📝 תיאור הפגיעות (טקסט חופשי):
"A buffer overflow in the login function of LibXYZ v2.3
 allows remote attackers to execute arbitrary code via
 a crafted HTTP request."
         │
         ▼
┌─────────────────────────────────────────┐
│          🤖 LLM ("הבוחן העיוור")         │
│                                         │
│  מה שהוא רואה: רק את הטקסט למעלה       │
│  מה שהוא לא רואה: CVE ID, CWE, CPE     │
│                                         │
│  המשימה: לנחש 8 ציונים בלי "לרמות"     │
│  (בלי לחפש את ה-CVE במאגר)             │
│                                         │
│  Output:                                │
│  AV=Network, AC=Low, PR=None, UI=None  │
│  S=Unchanged, C=High, I=High, A=High   │
└─────────────────────────────────────────┘
```

**נקודה קריטית:** המחקר **מוציא בכוונה** את ה-CVE ID מהפרומפט. למה? כי הכללת CVE ID מאפשרת למודל "לרמות" — לשלוף את הציון מזיכרון האימון במקום באמת לנתח את הטקסט. זו הנדסת מאפיינים גרועה (poor feature engineering) שהופכת את ההערכה ל-retrieval task במקום classification task.

### 3.3 מסד הנתונים

```
מקור: CVE List (MITRE) — https://github.com/CVEProject/cvelistV5
תקופה: 2019–2024 (CVSS v3.1 בלבד)
סינון: רק רשומות עם תיאור באנגלית + כל 8 מטריקות הבסיס
תוצאה: 31,000+ רשומות CVE
פורמט: JSON — כל רשומה כוללת תיאור + 8 מטריקות
```

**Ground Truth:** ציוני CVSS v3.1 מתוך CVE List של MITRE נחשבים כאמת הייחוס.

### 3.4 המודלים שנבדקו

שישה מודלי LLM, שילוב של מודלים סגורים ופתוחים:

| מודל | קיצור | סוג | פלטפורמה |
|------|--------|-----|----------|
| GPT-4o | G4 | סגור (Closed) | Azure AI Foundry |
| GPT-5 | G5 | סגור (Closed) | Azure AI Foundry |
| Llama-3.3-70B-Instruct | L | פתוח (Open) | Azure AI Foundry |
| Gemini-2.5-Flash | GM | סגור (Closed) | Google AI Studio |
| DeepSeek-R1 | DS | פתוח (Open) | Azure AI Foundry |
| Grok-3 | GR | פתוח (Open) | Azure AI Foundry |

**למה שילוב?** מודלים סגורים (כמו GPTs) מציעים דיוק ובטיחות; מודלים פתוחים (כמו Llama, DeepSeek) מציעים שקיפות, שחזוריות, וגמישות מחקרית.

### 3.5 הנדסת פרומפט (Prompt Engineering)

עיצוב הפרומפט בנוי בשני שלבים, לפי Best Practices של OpenAI:

```
┌──────────────────────────────────────────────────────┐
│                  Prompt Structure                     │
│                                                      │
│  System Message:                                     │
│  "Act as a cybersecurity expert.                     │
│   Given a CVE description, extract the 8 CVSS v3.1  │
│   base metrics. Return each metric individually."    │
│                                                      │
│  ⚠️ מה שה-Prompt לא מבקש:                           │
│  • לא מבקש ציון CVSS כולל (רק מטריקות בודדות)       │
│  • לא כולל CVE ID (למנוע "רמאות")                   │
│  • לא כולל CWE או CPE (רק תיאור טקסטואלי)          │
│                                                      │
│  Temperature: 0 (deterministic output)               │
│  Format: CSV structured output                       │
│  Batch size: 20 CVEs per API call                    │
└──────────────────────────────────────────────────────┘
```

**בחירת מספר ה-shots:** החוקרים בדקו zero-shot, two-shot, five-shot, ו-ten-shot על תת-קבוצה של 6,000 CVEs מ-2024. **two-shot** השיגה את הביצועים הטובים ביותר — ולכן אומצה לכל הניסויים.

## 4. מדדי הערכה (Evaluation Metrics)

### 4.1 הבעיה: חוסר איזון מחלקתי (Class Imbalance)

לפני שנבחן תוצאות — חשוב להבין שמסד הנתונים **לא מאוזן**. למשל, 83.8% מהפגיעויות מסווגות כ-Low Complexity:

| מטריקה | יחס חוסר איזון (Imbalance Ratio) |
|--------|----------------------------------|
| **Attack Vector** | **63.60** (הכי לא מאוזן!) |
| Attack Complexity | 5.19 |
| Privileges Required | 3.04 |
| Scope | 2.11 |
| Confidentiality Impact | 1.85 |
| Availability Impact | 1.77 |
| User Interaction | 1.72 |
| **Integrity Impact** | **1.23** (הכי מאוזן) |

**מטאפורה:** דמיינו שרופא צריך לזהות מחלות נדירות. אם 99% מהחולים הם בריאים, הוא יכול פשוט לומר "בריא" לכולם ולקבל דיוק 99% — אבל זה חסר ערך. לכן צריך מדדים שמתחשבים בחוסר האיזון.

### 4.2 חמישה מדדי הערכה

המחקר משתמש בחמישה מדדים משלימים:

**(1) Accuracy (דיוק כללי)**

```
Accuracy = # correct (y_i = ŷ_i) / N
```

מגבלה: מטעה כשיש class imbalance.

**(2) Weighted Precision (דיוק משוקלל)**

```
Weighted Precision = (1/N) × Σ(n_i × Precision_i)

כאשר: Precision_i = TP_i / (TP_i + FP_i)
```

**(3) Weighted Recall (שלמות משוקללת)**

```
Weighted Recall = (1/N) × Σ(n_i × Recall_i)

כאשר: Recall_i = TP_i / (TP_i + FN_i)
```

**(4) Weighted F1-Score**

```
F1_i = 2 × (Precision_i × Recall_i) / (Precision_i + Recall_i)

Weighted F1 = (1/N) × Σ(n_i × F1_i)
```

**(5) Mean Absolute Error (MAE — שגיאה ממוצעת מוחלטת)**

```
MAE = (1/N) × Σ|y_i - ŷ_i|
```

MAE רלוונטי למטריקות עם סדר פנימי (Ordinal), כמו Attack Vector (Network > Adjacent > Local > Physical). MAE לוכד לא רק אם הסיווג שגוי, אלא **כמה רחוק** הוא מהאמת.

- הפער הגדול ביותר: מטריקות **Impact** (קשות) מול מטריקות **Exploitability** (קלות)

## 6. מטא-מסווגים (Meta-LLM Classifiers) — שילוב מודלים

### 6.1 הרעיון

אם כל LLM טועה בצורה שונה, אולי **שילוב** של מספר LLMs יכול להפחית שגיאות? המחקר בנה מטא-מסווג שמקבל את הניבויים של ששת המודלים כ-input ומנסה לייצר ניבוי משופר.

### 6.2 מטאפורה: "חדר השופטים"

```
🏛️ חדר השופטים (Meta-Classifier)
    │
    ├── שופט 1 (GPT-5): "ההרשאות הנדרשות: LOW"
    ├── שופט 2 (GPT-4o): "ההרשאות הנדרשות: NONE"
    ├── שופט 3 (Gemini): "ההרשאות הנדרשות: LOW"
    ├── שופט 4 (Llama): "ההרשאות הנדרשות: HIGH"
    ├── שופט 5 (DeepSeek): "ההרשאות הנדרשות: NONE"
    ├── שופט 6 (Grok): "ההרשאות הנדרשות: LOW"
    │
    ▼
    🧮 מטא-מסווג: לא רק הצבעת רוב!
    מחשב:
    • ניבויים גולמיים (6 vectors)
    • הסכמה בין זוגות (pairwise agreement)
    • מחלקת הרוב (majority vote)
    • חוזק הקונצנזוס (confidence score)
    • דגלי אמינות (reliability flags)
    │
    ▼
    📋 הכרעה סופית: "LOW" (ברמת ביטחון גבוהה)
```

### 6.3 הנדסת מאפיינים (Feature Engineering)

המאפיינים שנכנסים למטא-מסווג:

1. **ניבויים קטגוריאליים** — הניבוי של כל אחד מ-6 המודלים, מקודד מספרית
2. **מדדי הסכמה** — אחוז הסכמה בין זוגות מודלים
3. **מחלקת הרוב** — על מה הרוב הצביע
4. **חוזק קונצנזוס** — כמה המודלים מסכימים (ציון אמון)
5. **דגלי אמינות** — האם כל מודל הפיק פלט תקין (לא UNKNOWN)

### 6.4 ארכיטקטורות מטא-מסווג שנבדקו

| ארכיטקטורה | תיאור |
|------------|--------|
| **Voting Ensemble** | הצבעה רכה — ממוצע התפלגויות הסתברות |
| **Random Forest** | עצי החלטה עם חשיבות מאפיינים |
| **Gradient Boosting** | תיקון שגיאות סדרתי (sequential error correction) |
| **Logistic Regression** | מודל לינארי עם מקדמים פרשניים |
| **SVM** | תמיכת וקטורים עם גרעין RBF |
| **Neural Network** | MLP עם שתי שכבות נסתרות (100, 50 נוירונים) + early stopping |

**הערכה:** 5-fold stratified cross-validation + חלוקת 80/20 train-test.

### 6.5 תוצאות המטא-מסווג

| מטריקת CVSS | Baseline | GPT-5 (הטוב ביותר) | מטא-מסווג (הטוב ביותר) | ארכיטקטורה | שיפור |
|-------------|----------|---------------------|------------------------|-----------|-------|
| Attack Complexity | 83.24 | **83.94** | **84.19** | LR | +0.24 |
| Attack Vector | 73.16 | 88.29 | **90.30** | RF | **+1.03** |
| Availability Impact | 39.41 | **67.08** | 67.37 | RF | +0.29 |
| Confidentiality Impact | 45.20 | **75.68** | **76.45** | SVM | +0.77 |
| Integrity Impact | 37.37 | **77.71** | **78.10** | NN | +0.39 |
| Privileges Required | 49.95 | **71.44** | 71.44 → 72.23 | SVM | +0.80 |
| Scope | 68.28 | 76.04 | **79.12** | NN | **+3.08** |
| User Interaction | 64.40 | **88.36** | **88.55** | SVM | +0.19 |

**ממצאים מרכזיים:**
- המטא-מסווג **שיפר את כל 8 המטריקות** ביחס למודל הטוב ביותר
- השיפור הגדול ביותר: **Scope (+3.08%)** — NN הצליח לשלב ידע ממספר מודלים
- השיפור הכולל: **79.54%** (מטא) מול **78.55%** (GPT-5)
- השיפור **צנוע** — מרבית השגיאות נובעות מ**חוסר מידע בתיאור** ולא מחוסר הסכמה בין מודלים

### 6.6 חפיפה בשגיאות בין מודלים

```
📊 חפיפת שגיאות סיווג (Misclassification Overlap):

  ████████████████████████████████ 28.7% → כל 6 המודלים טעו!

  ██████████████████ 17.9% → כל 6 המודלים טעו!
```

## 7. ניתוח איכות תיאורי הפגיעויות

### 7.1 אורך התיאור

```
📏 התפלגות אורך תיאורי CVE:

  אורך ממוצע: 361 תווים
  חציון:      271 תווים
  מקסימום:    3,810 תווים
  מינימום:    28 תווים

  Pearson r (אורך ↔ דיוק): ≈ 0 (אין קורלציה!)
```

**תובנה:** תיאור ארוך יותר **לא** מוביל לדיוק גבוה יותר. זה מראה ש-verbosity (פטפוט) אינו אותו דבר כמו informativeness (אינפורמטיביות).

### 7.2 ישויות בשם (Named Entities)

חילוץ ישויות ORG ו-PRODUCT באמצעות spaCy:

```
Pearson r (מספר ישויות ↔ דיוק): 0.048 (p < 0.05)
→ קורלציה חלשה עד זניחה
```

גם מספר הישויות (שמות ארגונים, מוצרים) **לא מנבא** דיוק. מה שחסר הוא לא שמות, אלא **פרטים טכניים** כמו: מהו וקטור ההתקפה, האם נדרשת אינטראקציית משתמש, מה הטווח.

### 7.3 תוכן מידע (Information Content)

חישוב IC (Information Content) באמצעות Semantic Correlation:

```
Pearson r (IC ↔ דיוק): אין קורלציה חזקה
```

**מסקנה:** אף אחד מהפרוקסיים (אורך, ישויות, IC) לא מנבא דיוק טוב. המשתנה המכריע הוא **האם התיאור מכיל את הפרטים הספציפיים** שנדרשים לכל מטריקה.

### 7.4 שדות נוספים (CPE, CWE, CVE ID)

| שדה נוסף | השפעה על הדיוק |
|----------|----------------|
| CPE (Common Platform Enumeration) | **שום שיפור** — המידע כבר בתיאור |
| CWE (Common Weakness Enumeration) | **שום שיפור** — לא מוסיף הקשר רלוונטי |
| CVE ID | **שיפור גדול אבל מטעה!** — המודל פשוט שולף מהזיכרון |

**תובנה קריטית:** הוספת CVE ID מייצרת שיפור מלאכותי. המודל לא "מסווג" — הוא "שולף" (retrieval). לכן ה-CVE ID הוצא מכל הניסויים, ו-CPE/CWE הוצאו גם כן כי הם מיותרים.

## 8. סיבות לשגיאות סיווג — מה ה-LLM אומר

החוקרים שאלו את GPT-4o שתי שאלות על מדגם של CVEs שסווגו לא נכון:
1. "למה סיווגת את ה-CVE הזה לא נכון?"
2. "איזה מידע נוסף היה עוזר?"

שלוש קבוצות סיבות:

### 8.1 מידע חסר (Missing Information)

```
❌ דוגמה לתיאור בעייתי:
"A vulnerability in Product X allows attackers to gain elevated privileges."

מה חסר:
  • האם ההתקפה דרך הרשת? מקומית? פיזית? → AV לא ברור
  • האם צריך אינטראקציה של המשתמש? → UI לא ברור
  • מהן ההרשאות הנדרשות? → PR לא ברור

→ ה-LLM מנחש default → בדרך כלל "Network, None, None"
```

### 8.2 שפה עמומה (Ambiguous Language)

```
❌ ניסוחים מטעים:
"arguments can be provided" → ה-LLM חושב: Low Complexity + No Privileges
                              בפועל: High Complexity + Low Privileges

"introduce properties"      → ה-LLM חושב: Low Complexity
                              בפועל: High Complexity (ניואנסים טכניים)
```

### 8.3 הטיית אימון (Training Bias)

```
❌ הכללת יתר מדוגמאות נפוצות:
ה-LLM ראה בעבר הרבה CVEs עם Attack Vector = NETWORK
→ נוטה לחזות NETWORK גם כשהתשובה היא LOCAL או PHYSICAL
→ over-generalization מדוגמאות שכיחות, שגיאה בנישה
```

## 9. מגבלות המחקר

### 9.1 מגבלות שהמחקר מזהה

1. **חפיפה עם נתוני אימון** — רשומות CVSS ישנות עלולות להופיע בקורפוס האימון של המודלים. המחקר הסיר CVE IDs כדי למנוע retrieval ישיר, אבל חשיפה חלקית אפשרית
2. **תלות בתיאור** — אם התיאור חסר, גם מודל מושלם ייכשל. הבעיה היא במקור הנתונים, לא רק במודל
3. **שיפור צנוע מהמטא-מסווג** — שיפור מקסימלי של +3.08% מראה שהצוואר-בקבוק הוא איכות המידע, לא ארכיטקטורת המודל
4. **חוסר Fine-tuning** — המחקר בדק רק zero/few-shot, בלי כוונון עדין ייעודי
5. **חוסר הקשר חיצוני** — לא שולבו ספריות, גרפי תלויות, או proof-of-concept

### 9.2 מגבלות שהמחקר לא מזהה (ביקורת נוספת)

1. **ביצועי baseline מלאכותיים** — Baseline של "majority class" הוא תחתון מאוד; כל שיטה סבירה תשפר עליו
2. **חוסר השוואה ל-ML קלאסי** — לא הושוו ל-BERT, XGBoost, או מודלים מסורתיים שנכתבו למשימה הזו ספציפית
3. **עלות** — עיבוד 31,000 CVEs דרך 6 APIs = אלפי קריאות API → עלות כספית משמעותית, לא תמיד מעשי
4. **Temporal validity** — המודלים אומנו על נתונים עד נקודת זמן מסוימת; פגיעויות חדשות עם מונחים חדשים עלולות לקבל ניקוד שגוי

## 11. רלוונטיות לתזה — ניקוד סיכון דינמי לסוכני MCP

### 11.1 מה הקשר הישיר?

המאמר הזה מדבר על ניקוד פגיעויות, אבל העקרונות ישירות רלוונטיים ל**ניקוד סיכון של סוכני MCP**:

```
המאמר הזה:                        התזה שלך:
┌──────────────────────┐          ┌──────────────────────────┐
│ Input: תיאור CVE     │          │ Input: בקשת גישה של סוכן │
│ Output: ציון CVSS    │          │ Output: ציון סיכון 1-10  │
│ (8 מטריקות בסיס)     │          │ (דינמי, מתעדכן)         │
│                      │          │                          │
│ שיטה: LLM קורא      │          │ שיטה: מערכת מעריכה       │
│ טקסט ומסווג          │          │ בקשה ומסווגת             │
│                      │          │                          │
│ אתגר: מידע חסר       │          │ אתגר: כוונה לא ברורה    │
│ בתיאור               │          │ של הסוכן                │
└──────────────────────┘          └──────────────────────────┘
```

### 11.2 לקחים ישירים לתזה

**לקח 1: אל תסמוך רק על ה-"תיאור" (Description)**

המאמר מראה בבירור ש-**תיאור טקסטואלי לבדו לא מספיק** לניקוד מדויק. זה ישירות רלוונטי ל-MCP:

```
CVE Description ≈ MCP Tool Description
  "A buffer overflow in..."    ≈    "get_weather: Returns weather data"

שניהם:
  ✅ נותנים רמז ראשוני
  ❌ לא מספיקים לניקוד מדויק
  ❌ יכולים להיות מטעים או עמומים
  ❌ חסרים הקשר (מי, איפה, עם מה)
```

עבור התזה: **הציון צריך להתבסס על הרבה יותר מה-tool description** — צריך את ההתנהגות בפועל, ההרשאות שנדרשות, הפרמטרים שנשלחים, ודפוסי שימוש היסטוריים.

**לקח 2: מטריקות Exploitability קלות יותר מ-Impact**

```
קלות לניבוי (Exploitability):          קשות לניבוי (Impact):
• Attack Vector     (F1=0.88)          • Confidentiality (F1=0.76)
• User Interaction  (F1=0.89)          • Integrity       (F1=0.78)
• Attack Complexity (F1=0.82)          • Availability    (F1=0.62)
```

עבור התזה: מטריקות **"איך" הסוכן ניגש** (דרך הרשת? מקומית? עם הרשאות?) יהיו **קלות יותר לניקוד** מאשר מטריקות **"מה ההשפעה"** (מה יקרה אם הסוכן ינצל לרעה?). זה אומר שכדאי **לתת משקל גבוה יותר** למטריקות ה-exploitability בניקוד הדינמי, כי הן יותר אמינות.

**לקח 3: Meta-Classifiers — שילוב מספר "שופטים"**

המאמר מראה ששילוב מודלים נותן שיפור **צנוע אך עקבי**. עבור ניקוד סוכני MCP:

```
במקום שופט אחד:
  LLM בודד → ציון סיכון = 7

שילוב "שופטים":
  מנתח הרשאות  → ציון 8    ┐
  מנתח התנהגות  → ציון 6    ├─→ ציון משולב = 7.2 (יותר אמין)
  מנתח הקשר    → ציון 7    │
  היסטוריה      → ציון 7.5  ┘
```

**לקח 4: Class Imbalance — רוב הבקשות יהיו תקינות**

בדיוק כמו ש-83.8% מה-CVEs הם Low Complexity, גם **רוב בקשות הגישה של סוכני MCP יהיו לגיטימיות**. מערכת הניקוד צריכה להתמודד עם חוסר האיזון הזה:

```
התפלגות צפויה של בקשות סוכן MCP:
  ✅ סיכון נמוך (1-3):  ~70% מהבקשות
  ⚠️ סיכון בינוני (4-6): ~20% מהבקשות
  🚨 סיכון גבוה (7-10):  ~10% מהבקשות

→ Accuracy לבדו לא מספיק! צריך Precision/Recall/F1 פר-מחלקה
→ חשוב במיוחד: Recall גבוה למחלקת "סיכון גבוה" (לא לפספס!)
```

**לקח 5: MAE — לא רק "נכון/לא נכון" אלא "כמה רחוק"**

המאמר משתמש ב-MAE כדי ללכוד את **מרחק השגיאה**, לא רק אם יש שגיאה. זה קריטי לתזה:

```
שגיאה של 1 נקודה:  ציון אמיתי 7, ניבוי 6  → MAE = 1 (סביר)
שגיאה של 5 נקודות: ציון אמיתי 9, ניבוי 4  → MAE = 5 (קטסטרופה!)

→ אם מערכת ניקוד הסיכון נותנת לסוכן מסוכן ציון 4 במקום 9,
   היא למעשה מרשה לו לפעול → פריצת אבטחה
```

### 11.3 מה לקחת ומה להשאיר

| מהמאמר | לתזה | הערה |
|--------|------|------|
| **לקחת:** מדדי הערכה (Weighted F1, MAE) | ✅ להשתמש באותם מדדים לניקוד סוכנים | מותאם מ-CVSS metrics ל-risk score |
| **לקחת:** ניתוח class imbalance | ✅ רוב הבקשות = סיכון נמוך | צריך stratified evaluation |
| **לקחת:** שילוב מסווגים (meta-classifier) | ✅ שילוב מספר אותות לציון אחד | SVM/RF עובדים טוב |
| **לקחת:** הסרת CVE ID (מניעת "רמאות") | ✅ לבדוק שהמודל לא מזהה סוכנים מוכרים | ניקוד צריך להיות מבוסס התנהגות |
| **להשאיר:** Two-shot prompting | ⚠️ למערכת real-time זה איטי | צריך fine-tuned model |
| **להשאיר:** ניתוח retrospective בלבד | ⚠️ המאמר בודק CVEs ישנים | התזה צריכה real-time scoring |
| **להשאיר:** הסתמכות על description בלבד | ❌ לא מספיק! | התזה צריכה גם: behavior, permissions, context |

### 11.4 רעיון ליישום: CVSS-inspired Agent Risk Scoring

בהשראת מבנה ה-CVSS, אפשר לבנות מערכת ניקוד דומה לסוכני MCP:

```
Agent Risk Score (1-10) = f(Exploitability, Impact)

Exploitability metrics (adapted for MCP agents):
  • Access Vector:      Network / Local / Sandboxed
  • Access Complexity:  Simple API call / Multi-step / Requires context
  • Privileges Needed:  None / Read-only / Read-Write / Admin
  • User Approval:      None / Notification / Explicit consent

Impact metrics (adapted for MCP agents):
  • Confidentiality:    Reads sensitive data? (None / Low / High)
  • Integrity:          Modifies data? (None / Low / High)
  • Availability:       Can cause downtime? (None / Low / High)
  • Scope:             Affects only target resource or beyond?

Agent Risk = 8.22 × AV × AC × PR × UA  ← exploitability subscore
           + Impact(C, I, A, S)          ← impact subscore
           + Behavioral Adjustment       ← dynamic modifier from history
```

זה **מרחיב** את ה-CVSS framework ממשהו סטטי (מנקד פגיעות חד-פעמית) למשהו **דינמי** (מנקד סוכן שפועל לאורך זמן, עם היסטוריית התנהגות).

## 12. שאלות פתוחות למחקר נוסף

1. **Fine-tuning:** האם כוונון עדין של LLM על 31,000 CVEs ישפר משמעותית? המאמר לא בדק
2. **הקשר חיצוני:** האם שילוב dependency graphs, PoC exploits, ו-vendor advisories ישפר את ניבוי ה-Impact?
3. **Temporal generalization:** האם מודל שאומן על CVEs מ-2019-2023 יעבוד טוב על CVEs מ-2025+?
4. **Cross-domain transfer:** האם העקרונות עובדים גם עבור ניקוד סיכון בדומיינים אחרים (MCP agents, API permissions, cloud access)?
5. **Human-in-the-loop:** האם שילוב LLM + אנליסט אנושי ייתן תוצאות טובות יותר משניהם לבד?

## סיכום ביניים

מאמר זה מספק ראיות אמפיריות ברורות ש-**LLMs יכולים לכמת פגיעויות אבטחה מתיאורים טקסטואליים**, אך עם מגבלות:
- ביצועים טובים על מטריקות "קלות" (Exploitability), חלשים על "קשות" (Impact)
- תיאורים עמומים או חסרים = צוואר הבקבוק, לא המודל
- שילוב מודלים עוזר, אבל בצורה צנועה
- **הרלוונטיות לתזה:** המבנה של CVSS (Exploitability + Impact) ניתן להתאמה לניקוד סיכון דינמי של סוכני MCP, תוך הרחבה מניתוח טקסטואלי סטטי לניתוח התנהגותי דינמי
